## 1. 系统简介

### 1.1 定位与目标

**叙事空间创作系统 v8.2**（Narrative Space Creation System）是一套面向番茄小说网男频商业网文的全流程自动化创作工具包。系统以 Claude Code Skill 形式交付，提供从作品立意到章节输出的完整闭环。

- **目标**：高质量、高爽感、商业化男频网文的批量生产
- **平台**：番茄小说网（番茄文风格，短句＋高信息密度）
- **技术栈**：Claude Code + MCP 工具（websearch、sequential-thinking）

### 1.2 核心能力

| 能力 | 描述 |
|------|------|
| 全流程规划 | 一句话创意 → 完整四页大纲，经过 12 步骤的系统性迭代 |
| 双模式写作 | 流水线模式（批量自动）+ 互动跑团模式（实时驾驶）|
| 动态知识库 | 全自动实体追踪、伏笔管理、角色弧光记录 |
| 八大创作法则 | 商业验证的剧情设计原则，内嵌于每次写作调用 |
| 质量双保险 | 8 维度评分 + Critic sub-agent 质检，确保章节合格率 |
| 去AI痕迹 | 专用 humanizer 后处理模块，消除 AI 文字痕迹 |
| **行文DNA提取** | 多作品联合提取人类写作 DNA，生成可执行风格约束（DO/DON’T + 标杆段落），写作时以最高优先级注入 |
| **TRPG沉浸互动** | 五种决策类型（行动/对话/态度/战术/内心）+ 决策密度自适应 + INTERRUPT紧急介入 |
| **记忆锚点防遗忘** | 每章 ≤150 字锚点（关键转折/悬念/情绪快照/下章债务），对抗长上下文遗忘 |

## 2. 快速开始

### 2.1 系统要求

- **Claude Code** CLI 或 VS Code Copilot Chat（支持 Skill 文件调用）
- **MCP 插件**（可选但推荐）：
  - `@modelcontextprotocol/server-brave-search` —— websearch 工具
  - `@modelcontextprotocol/server-sequential-thinking` —— sequential-thinking 工具

### 2.2 Skill 安装位置

本系统 SKILL.md 文件需放置在 Claude Code 能识别的 Skills 目录中：

```
~/.claude/skills/narrativespace-xushikj/SKILL.md
```

或工作区根目录下的 `.claude/skills/` 中。

### 2.3 初始化新项目

#### 三种运行模式

| 模式 | 命令 | 适用场景 |
|------|------|----------|
| **默认模式** | `python init.py --project-dir .` | 首次初始化；跳过已存在的文件，不覆盖任何数据 |
| **升级模式** | `python init.py --project-dir . --upgrade` | 已有项目升级系统版本；自动更新 `config/` 和 `references/`，**保留** `state.json`、`knowledge_base.json`、`memory.md` 等用户数据 |
| **强制模式** | `python init.py --project-dir . --force` | 重置项目；覆盖所有文件（包括用户数据）——**慎用** |
| **自动模式** | `python init.py --project-dir . --yes` | 跳过所有交互式确认，适合 CI / 脚本自动化场景 |

`--force` 与 `--upgrade` 不可同时使用。

> ⚠️ **Windows 用户注意：JSON 文件 BOM 问题**
>
> 若在 Windows PowerShell 中直接用 `Set-Content` 或重定向（`>`）写入 `state.json`、`knowledge_base.json` 等文件，PowerShell 默认会写入 **UTF-8 BOM（EF BB BF）**，导致 Python 读取时解析失败（`json.JSONDecodeError`）。
>
> **推荐解决方案**：
> - 始终通过 `init.py` 或系统脚本写入 JSON，脚本内部已做 BOM 自动剥除。
> - 手动编辑时使用 VS Code（另存为 UTF-8 不含 BOM）。
> - 如已出现 BOM，运行 `python scripts/init.py --project-dir . --upgrade` 将自动修复。

#### 缺失文件自动检测

每次运行时，脚本会在执行写入前**扫描项目中的缺失文件**，按三类列出：

```
[!] 发现 5 个缺失文件（本次运行将自动补充）:

  [用户数据] (1 个)
    - state.json

  [参考文件] (1 个)
    - references\few_shot_examples.md

  [配置文件] (3 个)
    - config\workflow.yaml
    - config\writing_rules.yaml
    - config\style_modules\fantasy.yaml

是否补充上述缺失文件？[Y/n]
```

- 输入 `Y`（或直接回车）：补充所有缺失文件后继续
- 输入 `n`：退出，不做任何修改
- `--force` 模式跳过询问，直接执行

#### 基本用法

```bash
# 进入你的小说项目目录
cd /path/to/your/novel-project

# 首次初始化
python /path/to/narrativespace-xushikj/scripts/init.py --project-dir .

# 升级已有项目（保留创作数据）
python /path/to/narrativespace-xushikj/scripts/init.py --project-dir . --upgrade
```

### 2.4 触发 Skill

在 Claude 对话中直接输入指令即可调用系统：

```
# 启动规划流程
用叙事空间系统帮我创作一部男频小说，先进行完整规划

# 直接开始写作（已有规划时）
用叙事空间系统继续写第 7 章，流水线模式

# 使用互动模式
用叙事空间系统启动跑团模式，我扮演主角林天

# 运行去AI痕迹处理
用叙事空间系统对以下文本进行 humanizer 处理：[粘贴文本]
```

### 2.5 首次使用推荐流程

```
1. 初始化项目目录（init.py）
2. Step 0（可选）：提供1-3部对标作品，系统学习风格
3. Step 1：输入你的核心创意概念（一句话）
4. Step 2：确定当前卷落点 + 全书司南
5. Step 2.5：确定世界观与金手指体系
6. Step 3：生成初始人物设定
7. Step 4：生成当前卷一页大纲
8. Step 5：生成人物弧光规划
9. Step 6：生成四页大纲（带伏笔）
10. Step 7：初始化知识库
11. Steps 8-9：规划场景
12. Step 10A 或 10B：开始写作
```

### 2.6 用户设定表单（user_settings_template.md）

`templates/user_settings_template.md` 是项目的**配置入口**，用于在对话开始时向 AI 声明核心参数。将此文件复制到你的项目目录，按需填写后粘贴到对话中。

**主要配置项**：

| 配置节 | 内容 |
|--------|------|
| 世界观与角色基础设定 | 对标作品、类型标签、世界背景、主角困境、金手指 |
| 扮演角色信息 | 角色名称、身份、性格、外貌、与用户关系 |
| 格式与输出设置 | 回复长度（A/B/C/D）、互动选项开关、规则自检等 |
| 特殊规则/状态管理 | 自定义追踪状态（好感度、爽感值等） |

**回复长度档位**：

| 档位 | 说明 |
|------|------|
| A | 每次 5000+ 字 |
| B | 每次约 4000 字 |
| C | 每次约 2500 字 |
| D（默认）| 自主决定，倾向长回复，最低 2000 字 |

## 5. 三层规划粒度系统

本系统使用三层嵌套的规划结构，平衡长期一致性与短期灵活性。

### 5.1 三层概述

| 层次 | 名称 | 调整幅度 | 锁定时机 |
|------|------|---------|---------|
| **第一层** | 长程司南（North Star）| 极小（<5%）| 第一卷规划时锁定 |
| **第二层** | 卷级大纲 | 中等（30%）| 提前2卷规划窗口 |
| **第三层** | 章节工作区 | 大（随时调整）| 前5章可完全改写 |

### 5.2 长程司南（第一层）

**定义**：全书的最终意境画面和核心信念，不绑定具体情节。

**正确示例**：
> 全书最终，林天站在废墟上看日出，他终于理解了父亲说的"力量是用来守护的"。

**错误示例**（太具体，会束缚中期创作）：
> 林天在第23卷第5章击败了大反派王朝并建立了新政权。

**司南的作用**：
- 每当规划新卷时，用司南校验：这一卷的发展方向是否朝着司南靠近？
- 当情节出现重大偏离时，用司南做仲裁

### 5.3 卷级大纲（第二层）

**规划策略**：
- 始终只提前规划 **当前卷+下一卷**
- 不对3卷后的情节做详细规划（"超期规划陷阱"）
- 每次开启新卷前，可输入**读者热力数据**来调整方向：
  - 章节热力图（哪些章节留存率高/低）
  - 读者评论摘要（哪些情节被讨论最多）
  - 跌落点分析（读者在哪章开始流失）

**卷级大纲可调节内容**（30%空间）：
- 调整子情节顺序
- 增删次要角色
- 改变某个反派的身份或阵营
- 调整爽感高潮的分布位置

### 5.4 章节工作区（第三层）

**定义**：当前正在写的章节 ± 前后5章的灵活调整空间。

**状态管理**：
- 写作完成的章节：**冻结**（不再修改，KB已录入）
- 当前写作的章节：**活跃**（可完全重写）
- 未来5章：**规划中**（有场景卡但可调整）

## 6. 八大创作法则

这是系统最核心的创作智慧，所有写作调用都内嵌这八条法则。

### 法则一：极限铺垫原则（四层困境）

**核心思路**：欲扬先抑——先把主角压到极限，再释放极限爽感。

**四层困境**（同时施压，不是选一个）：

| 层次 | 描述 | 示例 |
|------|------|------|
| **宏观困境** | 世界规则层面的压制 | 阶级制度、法律管控、天花板封印 |
| **中观困境** | 势力/团体层面的压制 | 门阀倾轧、组织背叛、多方围剿 |
| **微观困境** | 个人资源层面的压制 | 丹药耗尽、盟友背叛、情报误导 |
| **个人困境** | 心理/情感层面的压制 | 至亲受威胁、信仰崩塌、身份暴露 |

**执行要点**：
- 困境叠加的顺序：宏观→中观→微观→个人，逐层深入
- 每一层困境都必须看起来"无解"——读者要相信主角真的很惨
- 最后一层个人困境往往是"触发主角爆发"的导火索

---

### 法则二：期待感管理系统（信息差分层释放）

**核心思路**：不把信息一次性告诉读者，制造持续的"我想知道"欲望。

**三类钩子体系**：

1. **章末小钩子**（每章必须）：最后50-100字卡在关键动作之前，或揭示让读者意外的信息
2. **卷级大钩子**（每卷1-2个）：关乎主角命运的重大悬念，跨章维持期待
3. **全书超级钩子**（1-3个）：呼应长程司南，全书核心谜题

**信息差设计原则**：
- 让读者比主角**多知道一件事**（读者知道危险在哪，主角不知道）
- 或让读者比主角**少知道一件事**（主角有秘密，读者想猜）
- 两者交替使用，制造不同类型的紧张感

**九连环钩子设计（高级）**：
- 回收一个钩子的同时，埋下一个更大的钩子
- 确保读者始终有"还有一件事没解决"的感觉

---

### 法则三：连锁震惊反应（三层震惊）

**核心思路**：一件爽事发生时，让多个角色依次震惊，放大情绪涟漪。

**三层震惊结构**：

1. **直接执行者震惊**：被打脸的人的第一反应（不敢置信→愤怒/崩溃）
2. **周围亲友的震惊**：旁观者的反应（为主角欢呼/为反派叹气）
3. **权威人物的震惊**：高层/权威角色的惊讶（「连他也…？」）

**写法要点**：
- 三层震惊不要写成流水账，每层用不同的**感官细节**呈现
- 第三层权威震惊往往是最有爽感的：地位越高的人越惊讶，越爽
- 适当加入"旁白式总结"（类似弹幕效果：场内所有人都震惊了）

---

### 法则四：角色与剧情基因锚定（双轨弧光）

**核心思路**：男频小说有固定的男性基因驱动，所有角色设计都服务于此。

**主角行为准则**：主角的一切行为，其本质是"展示价值"，让世界（尤其是配角和对手）见证自己的强大。

**人物立体化三要素**：
1. **缺点设计**：主角必须有性格缺点（不是菜，是有执念或短板），这是成长的起点
2. **底层行为逻辑**：每个核心角色有一条"无论如何都不妥协"的底线，确保行为一致
3. **高概念生存动机**：主角的奋斗必须锚定一个有信仰底线的动机（家人安全、族群崛起、信念守护等）

**反派设计原则**：
- 反派的"恶"来自**立场冲突**，而非"天生坏人"
- 每个反派有专属的"攻击主角底线"的手段
- 反派智商必须在线——一个蠢反派提供不了真正的爽感

---

### 法则五：核心创作套路库

**解锁本法则**：在法则一到四的基础上，叠加以下经过商业验证的情节模式。

**套路一："锁"式危机设计**

- **时间锁**：设置倒计时压力（3分钟内/30天内/七天后），瞬间提升紧迫感
- **空间锁**：将角色限制在有限空间内（密室/孤岛/包围圈），强制制造冲突

**套路二："大小反差"设计**

- **大人物遇小事**：身份尊贵的主角处理日常小事，产生"降维打击"或"水土不服"的效果
- **小人物遇大事**：平凡主角卷入超规格事件，读者为其担忧和好奇

**套路三："道德挑战"设计**

- 设计主角在道德/利益/情感三者之间的两难选择
- 读者会极度投入地思考"我会怎么选"
- 主角的选择必须**符合角色底层逻辑**，而非随机的

---

### 法则六：数据化质量评估原则

系统的 8 维度质量评估，是法则六的实体化。见[第10节：质量评估系统](#10-质量评估系统)。

**核心优先级**：当质量评估中的维度与"爽感"产生冲突时，**爽感优先**。

---

### 法则七：高智商压迫与期待感拉扯

**核心公式**：没有极度憋屈，就没有极度爽快。

**反派设计要求（关键）**：

- 反派必须使用**规则杀（Rule Kill）**，而非纯暴力
  - 法规封口、阶层壁垒、信息管控、门阀制度……
  - 主角的胜利也必须是**对规则的瓦解**，而非单纯碾压
- 反派的计划必须**看起来天衣无缝**——读者要相信"这次主角真的赢不了"
- 反派智商不足→读者不觉得爽（因为赢得太容易）

**信息差陷阱设计**：

> 每隔5章必须设计一个"读者已知但主角未知"的信息陷阱。

类型：
- 反派的真实身份（读者知道，主角不知道）
- 即将到来的背叛（读者看到了迹象，主角还信任着那个人）
- 主角误解的关键线索
- 主角正在走入的埋伏

---

### 法则八：降维打击与错位震惊

**核心思路**：主角每次出手，必须打破常规认知——反派越自信，主角的反击越爽。

**降维打击四步结构**：

1. **反派布局完成**：反派认为已经赢定了，计划周密
2. **主角方形势最低落**：读者也开始为主角担忧（法则一的最深处）
3. **主角掀开金手指底牌**：用反派完全没料到的方式反击
4. **反派信仰崩塌**：反派的"怀疑人生"越深，读者越爽

**执行细节**：
- 降维打击不是"主角突然变强"——必须是**提前埋下的设定被激活**
- 反派的崩溃要写细：微表情、语气变化、动作细节（手抖、声音颤）
- 旁观者的"集体疑惑→震惊→服气"的过程要完整写出来

## 7. 流水线写作模式（Pipeline）

**适用场景**：稳定批量生产章节，无需用户即时介入。

### 7.1 架构总览

```
orchestrator (主调度进程)
  │
  ├─ 1. 读取 state.json + scene_plan
  │      提取当前章节的涉及角色IDs
  │
  ├─ 2. 切片 knowledge_base.json
  │      仅提取涉及角色/地点/事件的相关条目 → kb_slice
  │
  ├─ 3. 组装 instruction package
  │      = 场景卡 + kb_slice + 八大法则 + 风格规则 + 爽感类型
  │
  ├─ 4. 启动 chapter-writer sub-agent
  │      → 注入：情绪温度曲线 + 人味注入规则 + DNA约束（如已提取）
  │      → 输出：章节正文 + kb_diff（JSON格式）
  │
  ├─ 5. 应用 kb_diff
  │      → 增量更新 knowledge_base.json
  │
  ├─ 6. 启动 summary sub-agent
  │      → 生成章节概括（约150字）
  │      → 更新 summary_index.md（三级压缩策略）
  │
  ├─ 6.5. 生成记忆锚点（v8.0 新增）
  │      → 提取本章：关键转折 / 最紧迫悬念 / 主角情绪快照 / 下章债务
  │      → 写入 .xushikj/anchors/chapter_{N}_anchor.md（≤150字）
  │
  ├─ 7. 质量评估
  │      → 8维度评分 + DNA合规自检 Q8
  │      → 如 < 64分，触发 Critic sub-agent + 修订循环
  │      → 生成 quality_report
  │
  ├─ 8. 更新 state.json
  │      current_chapter += 1
  │      shuang_type_log 记录爽感类型
  │
  └─ 9. 向用户报告
         章节摘要 + 质量分数 + 下章预告
```

### 7.2 Chapter-Writer Sub-Agent 输出格式

chapter-writer 的输出是**双文件**：

**文件1：章节正文（Markdown）**
```markdown
# 第N章：[章名]

[正文内容，2000-4000字]
```

**文件2：KB差异文件（JSON）**
```json
{
  "chapter": "chapter_XX",
  "timestamp": "...",
  "add": {
    "entities": {
      "characters": {...},
      "locations": {...}
    },
    "foreshadowing": {...}
  },
  "update": {
    "entities.characters.char_001.arc_history": [...],
    "foreshadowing.planted.1.status": "resolved"
  }
}
```

### 7.3 Critic Sub-Agent（质量门控）

当某一章节质量评分 < 64分（合格线），自动触发 Critic sub-agent：

```
Critic sub-agent 流程：
  1. 读取原始章节 + 8维度评分详情
  2. 识别最低分维度（通常是 qd_07 钩子 或 qd_01 爽感）
  3. 检查 v8.0 红线四（DNA 偏离）：
     - 若 DNA 已提取，验证 DO 清单特征命中率 < 50% 或 DON'T 违规 ≥ 3 次
     - 红线四触发 → HALT，输出 DNA 违规报告，不进入修订循环
  4. 生成具体修订指令（精准操作而非泛泛建议）
  5. 将修订指令传回 chapter-writer 进行重写
  6. 重写后再次质量评估
  7. 最多循环 2 次（防止无限循环）
```

Critic 的 `tension_signal` 字段：
```json
{
  "tension_signal": {
    "current_level": 0.35,
    "trend": "falling",
    "recommendation": "立刻在场景中引入外部冲突"
  }
}
```

### 7.4 Cycle 收敛检查

每完成5章（一个cycle）后，自动执行收敛检查：

**检查项目**：
1. 爽感类型分布（5章中是否单调重复同一类爽感？）
2. 平均质量分数（本cycle均分 vs 上cycle均分）
3. 伏笔状态（新增伏笔数量 vs 回收伏笔数量，是否积压过多？）
4. 节奏分析（动作戏/对话戏/情感戏的比例）
5. 角色出场分布（是否某个重要角色消失太久？）

**收敛报告格式**：
```markdown
## Cycle 001 收敛报告

| 维度 | 本cycle | 目标 | 状态 |
|------|---------|------|------|
| 平均质量 | 68.2 | ≥64 | ✅ |
| 钩子命中率 | 4/5 | ≥4/5 | ✅ |
| 伏笔净植入 | +3 | ≤+5 | ✅ |
| 爽感类型多样性 | 3种 | ≥3种 | ✅ |
```

### 7.5 触发流水线写作

```
# 写单章
用叙事空间系统流水线写作第 [N] 章

# 写一个cycle（5章）
用叙事空间系统流水线写作 cycle 002，从第 [N] 章开始

# 继续写（自动读取state.json）
用叙事空间系统继续流水线写作
```

### 7.6 线热度连续性系统（Line Heat Continuity）

写作模块内置**线热度连续性规则**，防止长篇创作中故事线"消失"或"突然回收"。

**四类追踪对象**：

1. **角色线** — 核心与高频配角的出场节奏
2. **情节线** — 主线/支线的推进状态
3. **伏笔线** — 待回收的置入承诺
4. **算法标签线**（番茄特化）— 金手指热度、主线驱动力、书名关键词呼应频率

**热度状态**：

| 热度 | 含义 |
|------|------|
| `hot` | 当前正在驱动冲突或读者期待 |
| `warm` | 近期活跃，但非本章主焦点 |
| `cold` | 若再忽略将造成遗忘或断线风险 |
| `archived` | 已明确收束或长期休眠 |

**默认回顾阈值**：
- 核心角色：3-5章内需有有效触及
- 主要支线：4-6章内需有推进、转移或明确延迟
- 活跃伏笔：连续错过2次可回收窗口即判 overdue

**有效触及 vs 无效露面**：

| 类型 | 判定 |
|------|------|
| 直接推进该线 | ✅ 有效 |
| 压力转移并形成新义务 | ✅ 有效 |
| 给出可验证的延迟理由 | ✅ 有效 |
| 只点名不改变局面 | ❌ 无效 |
| 只做回忆性提及无后效 | ❌ 无效 |

**算法标签**（高追读率线条附加标注）：

| 标签 | 含义 |
|------|------|
| `#日更` | 本线近期驱动过连续更新承诺 |
| `#爆更点` | 本线即将触发大量信息差释放或连续爽点 |
| `#读者悬念` | 本线当前处于"读者已知而主角未知"状态 |
| `#深坑待填` | 本线有超过10章未回收的伏笔，进入高优先级 |

> `#深坑待填` 超过15章自动触发 WARN：伏笔遗忘风险。

**章后写回要求**（每章结束时记录）：
1. 本章触及了哪些线
2. 各线热度是否变化
3. `next_review_chapter` 是否需要前移
4. 是否产生 overdue 项

## 8. 互动跑团模式（Interactive）

**适用场景**：用户想实时扮演主角，对关键决策有掌控权的沉浸式创作体验。

### 8.1 角色分工

| 角色 | 由谁扮演 | 职责 |
|------|---------|------|
| **DM（地下城主）** | AI | 世界呈现、环境描述、NPC 言行、事件触发 |
| **玩家（主角）** | 用户 | 主角的关键决策、对话内容、行动选择 |

### 8.2 状态机（全流程）

```
INIT ──→ OPENING ──→ PING_PONG ←──→ ROLLBACK
                         │  ↑
                         │  └──── INTERRUPT（v8.0 新增）
                         │
                         ↓ (节奏报警)
                   PACING_ALERT
                         │
                         ↓ (手动或触发落盘)
                      LANDING ──→ MAINTENANCE
```

**各状态说明**：

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| **INIT** | 初始化状态 | 用户启动跑团模式 |
| **OPENING** | 开局叙述 | INIT 完成后 |
| **PING_PONG** | 主循环：DM描述+玩家行动 | OPENING 完成后 |
| **ROLLBACK** | 回滚到上一个决策点 | 用户输入"撤销"/"重来" |
| **INTERRUPT** | 即时干预当前生成方向（v8.0）| 用户输入"等等"/"停"/"我要插嘴" |
| **PACING_ALERT** | 节奏警告 | 连续3回合无实质推进 |
| **LANDING** | 本章落盘（保存）| 用户输入"落盘"或达到字数上限 |
| **MAINTENANCE** | 维护/KB同步 | LANDING 完成后 |

### 8.3 PING_PONG 主循环

**决策密度自适应（v8.0）**：DM 根据当前的 `scene_pressure`（场景压力值）自动切换输出密度：

| 密度档位 | 触发条件（scene_pressure）| 片段长度 | 典型场景 |
|---------|--------------------------|----------|---------|
| **dense（密集）** | ≥ 8 | 100-200字 | 战斗/对峙/生死抉择 |
| **normal（常规）** | 4-7 | 300-600字 | 社交/探索/轻度冲突 |
| **sparse（宽松）** | < 4 | 800-1200字 | 过渡/赶路/休息场景 |

用户也可手动覆盖：`/密度 dense` | `/密度 normal` | `/密度 sparse`

每一回合的标准格式：

**DM 输出（AI）**：
```
---
[环境/状态更新：2-3行，简洁]

[场景核心描述：按决策密度控制字数，番茄文风格：短句、画面感]

[NPC 言行（如有）]

[决策选项]：（根据场景类型自动选择决策类型）
A. [选项A - 含风险预览标签]
B. [选项B - 含风险预览标签]
C. 自定义行动
---
```

**五种决策类型（v8.0）**：

| 决策类型 | 图标 | 适用场景 | 选项格式 |
|---------|------|---------|---------|
| **行动决策** | ⚔️ | 默认场景（攻击/逃跑/搜索）| `[选项A - {行动标签}]：{行动描述}` |
| **对话决策** | 💬 | NPC 抛出问题/条件/挑衅 | `[选项A - {语气标签}]：「{台词原文}」` |
| **态度决策** | 🎭 | 社交场合需要表态 | `[选项A - {态度}]：{表现描述}` |
| **战术微操** | 🎯 | dense 密度战斗中 | `[选项A - {战术}]：{操作与预期效果}` |
| **内心决策** | 💭 | 主角受重大打击/背叛/意外 | `[选项A - {心理方向}]：{内心变化及暗示}` |

**风险预览标签（consequence_preview）**：每个选项附带后果风险提示：
- 🟢 低风险：基本可控，失败代价小
- 🟡 中风险：可能引发连锁反应或失去某些资源
- 🔴 高风险：可能彻底改变局面，后果不可逆

**玩家输入（用户）**：
- 选择选项（A/B/C）
- 或直接输入自定义行动

**每回合字数目标**：300-600字（DM部分），不超过600字，保持快节奏

### 8.4 ROLLBACK（回滚机制）

当玩家对某个回合结果不满意：

```
用户输入：撤销 / 重来 / rollback
```

系统执行：
1. 恢复到上一个决策点的状态
2. 保留KB中已确认的实体（不回滚）
3. 重新提供该决策点的选项（可增加新选项）

**回滚限制**：最多回滚3次，超过后提示用户"本段剧情已锁定"。

### 8.5 LANDING（落盘）

落盘 = 本次跑团的成果保存为正式章节文件。

**触发方式**：
```
用户输入：落盘 / 结束本章 / landing
```

**落盘流程**：
1. DM 整理本回合所有片段
2. 生成完整章节正文（修复格式、补全场景衔接）
3. 生成 kb_diff（记录本章的实体变更）
4. 应用 kb_diff 到 knowledge_base.json
5. 生成章节概括，更新 summary_index.md
6. **生成记忆锚点（v8.0 新增）**：提取本章 ≤150字锚点，写入 `.xushikj/anchors/chapter_{N}_anchor.md`
7. 更新 state.json
8. 生成质量报告

### 8.6 PACING_ALERT（节奏警告）

触发条件：
- 连续3回合玩家只进行对话，无实质行动推进
- 连续3回合场景没有发生变化
- 累计字数超过3000字但剧情推进不足0.5个场景卡节点

警告格式：
```
⚠️ 节奏提醒：当前剧情推进较慢。
建议：引入一个外部事件（袭击/消息/意外），
或选择"跳过"当前互动，直接推进到下一个关键节点。
```

### 8.7 INTERRUPT（即时干预，v8.0 新增）

与 ROLLBACK 的区别：
- **ROLLBACK**：事后回退已落笔的片段
- **INTERRUPT**：在 DM 产出方向时实时改变走向（原片段不入库）

**触发词**：`等等` / `停` / `我要插嘴` / `换个方向`

**用法示例**：
```
用户：等等，我不想这么做。改成直接拔剑。
→ DM 丢弃当前生成，以新指令重新产出
```

**约束**：
- INTERRUPT 不消耗回合数（同一回合的重新生成）
- 可连续触发，无次数上限（与 ROLLBACK 的3次限制不同）

### 8.8 重锚周期（v8.0 新增）

**每5回合**（或累计字数 > 3000 字时）自动执行重锚：

```
步骤 1：重新读取 state.json → 提取关键状态
步骤 2：按 required_context_files 清单重载配置
步骤 3：重新读取最近 3 章记忆锚点
步骤 4：输出确认：[重锚完成 | 回合{N} | 规则已重激活]
```

> 重锚期间暂停 PING_PONG，完成后自动恢复。作用：对抗长对话中的注意力衰减（AI 遗忘）。

### 8.9 互动模式触发

```
# 启动互动模式
用叙事空间系统启动跑团模式，我扮演主角[名字]

# 指定从某章开始
用叙事空间系统跑团模式从第[N]章的场景开始

# 落盘保存
落盘

# 回滚（撤销已落笔片段）
撤销 / 重来

# 即时干预（改变当前方向）
等等 / 停 / 换个方向
```

- 钩子与下章内容相关性弱

### 11.1 系统说明

帮回系统是**独立于写作模式**的即时辅助机制：无论当前在流水线模式还是互动模式，用户随时可以输入帮回指令获得叙事策略支持。

| 指令 | 类别 | 效果 |

## 12. 去AI痕迹后处理（Humanizer）

### 12.1 核心原则

```
只做减法和替换，不加新内容。
保护小说叙事：象征、隐喻、心理描写、破折号都是合法文学工具，不动。
宁可漏改，不可误伤。
```

### 12.2 规则一：「不是…而是…」句式清除

**扫描目标**（所有否定+转折对比句）：
- `不是A，而是B`
- `不是A，是B`
- `不仅是A，更是B`
- `与其说是A，不如说是B`

**清除策略（99%删除率）**：

只有当同时满足以下两条件时才保留原句：
1. 删除会导致剧情逻辑断裂（不是修辞问题，是信息缺失）
2. 对比双方存在读者无法自行推导的信息差

**四种重写方式**（按优先级）：

| 方式 | 示例 |
|------|------|
| 转为单肯定 | `不是勇敢，而是绝望` → `那是绝望` |
| 转为双肯定 | `不是冷漠，而是克制` → `是克制，也是自我保护` |
| 转为具体描写 | `不是恐惧，而是敬畏` → `他的膝盖不自觉地弯了下去` |
| 直接保留B | `不是风声，而是哭声` → `那是哭声` |

### 12.3 规则二：稀疏排版/极短句清除

**扫描目标**：
- 独占一行且 ≤15 字的非对话句子
- 连续 2+ 个短句各占一行（诗歌体排版）
- 段落只有 1-2 句话（非场景转换标记）

**处理策略**：
- 合并到前段末尾，用逗号/句号衔接
- 语义独立的短句：重写为完整句子融入上下文

**例外（不动）**：
- 对话中的自然停顿
- 特殊格式（章节标题、节奏极快的打斗动作）
- 场景切换标记（`——三天后——`）

### 12.4 规则三：高频AI词汇替换

核心替换表（部分示例）：

| 原文（AI痕迹）| 替换方向 |
|------------|---------|
| 此刻 | 这一刻 / 现在 / 当下 |
| 然而 | 但 / 只是 / 偏偏 |
| 同时 | 顺带 / 顺手 / 一边……一边…… |
| 不仅如此 | 删除，直接说后半句 |
| 令人窒息 | 他胸腔发紧 / 气堵在喉咙里 |
| 从而 | 直接写结果，删掉连接词 |
| 其实 | 尽量删除或改为具体化陈述 |

### 12.5 规则四（v8.0 新增）：DNA回归校验（R-DNA）

**触发条件**：仅在项目已完成行文DNA提取（存在 `dna_human_*.yaml`）时执行。

**校验逻辑**：
```
在完成规则一至三的清除替换后，对处理结果执行反向扫描：

检查点 1：DO 清单特征保留率
  → DNA 的 DO 清单中，有多少特征仍然出现在处理后的文本？
  → 若 DO 特征命中率 < 70%（即超过30%的DNA特征被意外清除）
  → WARNING：过度清洗可能已损伤风格DNA，建议人工复核

检查点 2：主动保护
  → 若 DO 清单中的特征（如特定句式节奏/感官密度要求）
    与规则一至三的清除目标产生冲突——以 DO 清单优先
  → 宁可保留该特征，不因"可能是AI痕迹"而清除
```

**核心原则**：humanizer 的目的是清除 AI 的通病模式，而不是消除作者的个人风格。DNA 校验防止矫枉过正。

**Humanizer 的操作方式**：

```
# 对一段文本进行处理
用叙事空间系统对以下文本去AI痕迹处理：
[粘贴文本]

# 对整章进行处理
用叙事空间系统对第[N]章进行 humanizer 处理
```

## 16. 常见问题FAQ

### Q1：没有安装 MCP 工具，系统还能用吗？

**可以**。MCP 工具是增强功能，非必须。降级处理：
- Step 0 对标分析：跳过，或用户手动提供对标作品的分析资料
- Step 2.5 世界观：由用户直接输入世界观设计，系统不调用 websearch
- 大纲推演（Steps 4/6/8-9）：使用内置的线性推演替代，深度略低

确保 `.xushikj/state.json` 的 `writing_state.current_chapter` 已更新到正确的章节号，然后：

```
用叙事空间系统继续写作，从第[N+1]章开始
```

系统会自动读取 state.json 和 summary_index.md 恢复上下文。

### Q3：如何开始第二卷？

```
用叙事空间系统开启第2卷规划
```

系统自动执行：
1. `state.json.planning_state.current_volume` → 2
2. `state.json.planning_state.current_step` → 回退到 Step 4（卷大纲规划）
3. 提示用户：是否输入第一卷的读者热力数据来调整第二卷方向？

### Q4：quality_report 显示某章得分很低怎么办？

系统会自动触发 Critic sub-agent 进行修订。如果自动修订后仍不合格：

```
用叙事空间系统重写第[N]章，重点提升 qd_07（钩子）
```

手动指定要提升的维度，系统会针对性地重写该部分。

### Q5：跑团模式落盘后如何切回流水线模式？

```
用叙事空间系统切换到流水线模式，从下一章开始
```

系统更新 `state.json.writing_mode` 为 `pipeline`，并基于落盘的章节内容更新 scene_plan。

### Q6：伏笔积压太多（pending 数量超过 10 个）怎么办？

触发帮回指令：
```
帮回爽点分析
```

或手动触发伏笔回收规划：
```
用叙事空间系统分析当前积压伏笔，规划最近5章的回收顺序
```

### Q7：角色行为出现前后矛盾怎么办？

```
帮回人设检查 [角色名]
```

系统会对比该角色的初始设定（Step 3）、KB 中的 arc_history 和最近的实际写作内容，找出矛盾点并提供修复建议。

### Q8：生成的文本 AI 感很强，如何处理？

```
用叙事空间系统对第[N]章进行 humanizer 处理
```

或粘贴具体文本段落请求处理。Humanizer 专门针对番茄文的 AI 痕迹清除，不会改动场景结构。

### Q9：想临时更改世界观或金手指设定怎么办？

1. 更新 `.xushikj/outline/worldview_and_system.md`
2. 更新 `.xushikj/knowledge_base.json` 中受影响的 abilities 实体

## 24. RAG 本地语义检索（v8.2）

### 24.1 功能概述

v8.2 新增 **本地 RAG（Retrieval-Augmented Generation）** 向量检索引擎。当项目写到 30 章以上时，Orchestrator 会在每章写作前自动执行语义检索，找出与当前场景**情节最相关**的历史章节片段，注入 `semantic_memory_snippets` 字段，帮助 AI 保持跨章节一致性，避免"遗忘梗"和"情节矛盾"。

**核心特性**：
- 零 pip 依赖（只用 Python 标准库 urllib + math）
- 向量模型：Ollama `nomic-embed-text`（本地运行，隐私保护）
- 存储：每个项目独立 `.xushikj/rag/rag_index.json`（~3KB/章），不跨项目共享
- 四层实体隔离锁，防角色"串文"污染

### 24.2 三级降级链

| 级别 | 条件 | 行为 |
|------|------|------|
| **L1** | Ollama 可达 + nomic-embed-text 已拉取 | 真实余弦相似度检索 |
| **L2** | Ollama 不可达，但 retrieval_index.md 存在 | LLM 内联排序（文本关键词匹配） |
| **L3** | 以上均不可用 | 静默跳过，不阻塞写作 |

L2 降级时系统仍能工作，只是检索精度略低。

### 24.3 快速启动

**前置要求**（仅 L1 模式需要）：

```bash
# 1. 安装 Ollama（https://ollama.ai）
# 2. 拉取嵌入模型
ollama pull nomic-embed-text

# 3. 确认 Ollama 运行
ollama serve
```

**初始化并构建索引**（项目已写 N 章后执行一次）：

```bash
# 方式一：通过 init.py 初始化时附带构建
python scripts/init.py --project-dir /path/to/project --upgrade --build-rag-index

# 方式二：手动构建
python scripts/rag_index.py --project-dir /path/to/project --build
```

**检查后端状态**：

```bash
python scripts/rag_index.py --project-dir /path/to/project --check-backend
```

输出示例：
```json
{"backend": "ollama", "model": "nomic-embed-text", "note": "Ollama L1 就绪"}
```

**每章完成后增量更新**（Orchestrator 自动执行，也可手动）：

```bash
python scripts/rag_index.py --project-dir /path/to/project --add-chapter 35
```

**手动查询测试**：

```bash
python scripts/rag_index.py --project-dir /path/to/project \
  --query "主角对老师情感变化" \
  --char-ids char_001,char_003 \
  --current-chapter 36 \
  --top-k 3
```

**检查索引新鲜度**：

```bash
python scripts/rag_index.py --project-dir /path/to/project --check-staleness
```

### 24.4 四层实体隔离锁

RAG 检索的核心问题是"串文"——将 A 角色的回忆片段错误关联到 B 角色场景。v8.2 通过四层机制防止：

| 层 | 机制 | 说明 |
|----|------|------|
| **L1** | 硬实体过滤 | 章节角色 ID 与查询角色 ID 零交集 → 直接排除 |
| **L2** | 时效排除 | 最近 2 章已由 recent_summaries 覆盖，不重复注入 |
| **L3** | 余弦评分 | 相似度最高的条目优先 |
| **L4** | 多样性上限 | 每个主角色最多贡献 `max(2, top_k//2)` 条 |

### 24.5 索引文件说明

`.xushikj/rag/` 目录包含：

| 文件 | 大小参考 | 说明 |
|------|---------|------|
| `rag_index.json` | ~3KB/章 | 向量索引（含 768 维浮点嵌入） |
| `retrieval_index.md` | ~80字/章 | 人类可读索引，L2 降级时使用 |

200 章后 `rag_index.json` 约 620KB，对项目存储无明显影响。

### 24.6 触发条件与自动化

- **自动触发**：`state.json → rag_state.total_indexed > 30`（30 章前跳过）
- **每章后自动更新**：Orchestrator 在写完每章后调用 `--add-chapter N`（需在 memory.md 中启用 RAG）
- **关闭 RAG**：将 `state.json → rag_state.enabled` 设为 `false` 即可停用

### Q13：情绪温度和叙事张力（tension）有什么区别？

两者正交（互不干扰）：
- **tension（张力）**：控制"读者有多紧张"—— 剧情结构层面，决定悬念密度
- **temperature（温度）**：控制"文字有多少人味"—— 文字质感层面，决定句式和感官密度

例如：
- **高张力 + 低温度** = 冷酷智斗（紧张但克制，如权谋分析场景）
- **高张力 + 高温度** = 热血决战（既紧张又情绪滚烫）
- **低张力 + 高温度** = 温馨日常（放松但有人情味）

详见 [第20节：情绪温度系统](#20-情绪温度系统v80)

### Q14：DNA约束会不会让写出来的文字太像某个特定作者？

DNA 约束的设计目标是提取**跨作者的共性人类写作语感**，而非复制某个作者的风格。

具体防护机制：
- 联合提取要求 **2部以上不同作品**，归纳共性特征（排除个人风格噪声）
- DO 清单优先选取"在人类写作中普遍存在、AI 文本中稀缺"的特征
- DON'T 清单主要针对 AI 的通病写法，不针对某个作者的个人习惯

## 26. 长篇连载推荐工作流（v8.2）

> 适用场景：目标篇幅 ≥ 100 章、多卷结构的商业网文连载项目。

### 26.1 分卷快照机制

每一卷写完后，运行 **volume_snapshot.py** 生成卷末快照，自动提取：
- 存活角色状态（`knowledge_base.json` 中 `status ≠ 死亡` 的角色）
- 当前弧光阶段（`arc_stage` 字段）
- 待兑现伏笔（`foreshadowing.planted`，`status=pending`）
- 当前卷情节总结（从 `summaries/_index.md` 提取）


## 19. 行文DNA提取系统（v8.0）

> **作用**：解决 AI 生成文字缺乏"人类作者个人语感"的根本问题。通过多作品联合分析，提取超越单个作者的共性人类写作 DNA，让写作引擎在生成阶段就以最高优先级匹配目标语感。

### 19.1 触发条件

| 场景 | 参考文本量 | 执行流程 |
|------|-----------|---------|
| 仅 1 部作品 | ≥ 500字 | 标准逆向工程流程（生成 `clone_{name}.yaml`）|
| **2 部以上作品** | 每部 ≥ 500字 | **DNA 联合提取**（生成 `dna_human_{name}.yaml`）|

在 Step 0 完成分析后，系统会主动询问是否开启逆向工程模式（v8.0 默认推荐）。

### 19.2 DNA 提取流程

**Step DNA-1：逐作品特征卡**

对每部参考作品独立提取：

```yaml
work_name: "作品A"
sentence_rhythm:
  avg_sentence_length: 22
  short_long_ratio: "3:2"
  rhythm_pattern: "短-短-长-短（三短一长为主节奏）"
vocabulary:
  high_freq_words: ["XX", "XX", ...]
  avoided_words: ["XX", "XX", ...]
dialogue:
  avg_turn_length: 18
  density_per_1000: 45
description:
  sensory_per_1000: 8
  non_visual_ratio: 0.4
```

**Step DNA-2：跨作品共性归纳**

横向对比所有特征卡，提取一致性规律（标注置信度 high/medium/low）。

**Step DNA-3：输出 DNA 约束文件**

生成 `.xushikj/config/style_modules/dna_human_{project_name}.yaml`：

```yaml
# dna_human_林天传.yaml
project: "林天传"
source_works: ["赘婿", "大奉打更人"]

do_list:  # 必须在写作中复现的语感特征
  - id: "do_01"
    rule: "动词前置——行动句主语多省略，直接以动词开头"
    example: "拔出腰刀。转身。眼神扫过四周。"
  - id: "do_02"
    rule: "每300字至少一个身体感知细节（非视觉优先）"
    example: "背脊发凉。/喉咙里涌上腥味。/脚踝僵住了。"

dont_list:  # 此风格中绝不出现的写法
  - id: "dont_01"
    rule: "禁止使用抽象评论词作为情绪替代品"
    example: "BAD: 令人窒息的沉默 → GOOD: 他的手抖了一下"
  - id: "dont_02"
    rule: "对话中禁止出现'不是A，而是B'句式（已由 humanizer 统一覆盖）"

benchmark_paragraphs:  # 5-8条标杆段落，few-shot 示例直接引用
  - "【动作场景范例】卷腹起身。刀风从耳廓贴过。三步，四步——踏上石台，俯身压住了她的肩膀。"
```

### 19.3 写作引擎注入机制

DNA 文件注册到 `style_modules/index.yaml` 后，写作引擎在每次生成时：
1. 加载 DO 清单作为**最高优先级**写作约束（高于 style_rules.yaml）
2. 加载 DON'T 清单作为**最高优先级**禁止规则
3. 随机抽取 2-3 条 benchmark_paragraphs 作为 few-shot 示例注入 prompt

### 19.4 自检规则（v8.0 self_check Q8）

章节生成完成后，chapter-writer 自动执行 DNA 合规检查：
- **dna_compliance**：DO 特征命中率 ≥ 50% 且 DON'T 违规 ≤ 2 次 → 合格
- 若不合格：HALT，输出 DNA 违规报告（而非静默通过）

## 20. 情绪温度系统（v8.0）

> **作用**：独立于叙事张力（tension）的文字质感控制系统。tension 控制"读者有多紧张"；temperature 控制"文字有多少人味和热度"。两者正交，互不干扰。

### 20.1 三档温度定义

| 温度档 | 值域 | 名称 | 适用场景 |
|-------|------|------|---------|
| **cold（冷）** | 1-3 | 冷叙事 | 智斗/推理/商战博弈/冷酷复仇 |
| **warm（暖）** | 4-6 | 暖叙事 | 角色日常/信任建立/成长反思 |
| **hot（滚烫）** | 7-10 | 滚烫叙事 | 愤怒爆发/生离死别/极致爽感/决死 |

### 20.2 各档写作效果

**冷叙事（1-3）**：
- 句式：以陈述句和短句为主，少用感叹号
- 词汇：精确、冷硬的动词名词，减少情绪形容词
- 感官密度：低——仅保留功能性感官描写（位置/距离/光暗）
- 叙述距离：远——旁观者视角，不深入角色内心
- 对话风格：精炼、机锋、信息密度高、潜台词重

**暖叙事（4-6）**：
- 句式：长短句交错，允许偶尔散文化
- 词汇：允许更多感性词汇，加入具身体验（温度/气味/触感）
- 感官密度：中——每300字至少一个非视觉感官细节
- 叙述距离：中——可短暂进入角色内心展示真实想法
- 对话风格：允许废话、口头禅、不完整句、跑题（体现真实人际交流）

**滚烫叙事（7-10）**：
- 句式：极端节奏——短到只有几个字的爆裂句与长到一口气读完的奔涌句交替
- 词汇：允许极致修辞、夸张比喻、感官过载（必须具身化，禁止抽象）
- 感官密度：高——每100字至少一个感官冲击（痛感/温度/声音/气味）
- 叙述距离：近——几乎意识流，读者与角色共呼吸
- 对话风格：爆裂的、断裂的——人在极端情绪下不会好好说话

### 20.3 温度曲线（章节级规划）

在 `chapter_control_card` 中声明温度变化节点：

```yaml
emotional_temperature_curve:
  - position: "opening"
    temperature: 4
    reason: "日常对话建立信任"
  - position: "conflict_peak"
    temperature: 8
    reason: "背叛揭露"
  - position: "closing"
    temperature: 3
    reason: "冷静离开，伏笔"
```

### 20.4 使用方式

DM/chapter-writer 自动按温度曲线调整写法。用户也可手动指定：
```
用叙事空间系统以情绪温度7写作这段生死决战
用叙事空间系统用冷叙事风格（温度3）写林天计划复仇的独白
```

## 21. 人味注入规则（v8.0）

> **作用**：在生成阶段就植入人类写作的"生命痕迹"，而非仅依赖事后 humanizer 精洗。六条规则编入 write_constraints，每章/每回合生效。

### 21.1 规则总览

| 规则ID | 名称 | 频率要求 | 优先级 |
|--------|------|---------|--------|
| `ht_01` | 不完美注入 | 每2000字≥1次 | 中（不破坏爽感）|
| `ht_02` | 节奏呼吸 | 连续3段快节奏后触发 | 中高 |
| `ht_03` | 感官锚点 | 每500字≥1次感官细节 | 高 |
| `ht_04` | 内心独白 | 每1000字≥1次 | 中 |
| `ht_05` | 微矛盾 | 每2000字≥1次 | 低（不干扰情节逻辑）|
| `ht_06` | 叙事距离变换 | 每章至少1次视角拉近/推远 | 中 |

### 21.2 规则详解

**ht_01 不完美注入**：主角偶尔自嘲、对话中出现废话/打断/改口：
```
「我……不，算了。」他吞回了那句话。
（他忽然觉得自己像个傻子。）
```
> 禁止：在紧张度 > 8 的生死场景中出现轻浮自嘲

**ht_02 节奏呼吸**：连续3段快节奏（平均句长 < 15字）后，强制插入一段慢速感官段落（50-100字）：
```
（连续三段激烈战斗后——）
风停了。
他半跪在碎石堆里，耳鸣像有人在脑子里拉小提琴。
嘴里全是血的铁锈味。天空很蓝。蓝得不像是刚打完仗的天。
```

**ht_03 感官锚点**：每500字至少一个非视觉感官细节（优先级：触觉 > 听觉 > 嗅觉 > 味觉 > 视觉）：
```
背脊发凉。/膝盖僵住了。/隔着布料感到他手心的热度。
```

**ht_04 内心独白**：每1000字至少一次对外显动作有出入的内心活动：
```
外表：他点了点头，「行。」
内心：（行什么行。他不行。他只是暂时没办法说不行。）
```

**ht_05 微矛盾**：主角偶尔出现轻微的前后矛盾（如说了某话但行动不符），不影响情节逻辑，仅增真实感：
> 限制：每章至多1次；不得用于推进情节的关键台词

**ht_06 叙事距离变换**：每章至少1次在"远距离旁观"与"近距离意识流"之间切换，制造镜头感。

## 22. 记忆锚点系统（v8.0）

> **作用**：对抗 AI 在长篇创作中的"上下文遗忘"问题。每章提炼 ≤150字的极度压缩锚点，在后续章节开始时优先加载，确保 AI 始终记得"刚刚发生了什么"。

### 22.1 锚点格式

每章落盘后，系统自动生成 `.xushikj/anchors/chapter_{N}_anchor.md`：

```markdown
# 第N章 记忆锚点

**关键转折**：[本章最重大的情节转变，1-2句]
**最紧迫悬念**：[读者最想知道答案的问题，1句]
**主角情绪快照**：[主角当前最突出的情绪状态，10字以内]
**下章债务**：[本章结尾制造的、下章必须兑现的承诺，1句]

---
*总字数：≤150字（含标签）*
```

**示例**：

```markdown
# 第7章 记忆锚点

**关键转折**：林天发现三叔父才是灭门幕后的真正指使者，但他没有揭穿。
**最紧迫悬念**：林天为什么选择沉默？他在等什么时机？
**主角情绪快照**：表面冷静，内心崩溃后的空白
**下章债务**：林天在宗祠外独自坐到天亮，需要在下章开头出现

---
*总字数：86字*
```

### 22.2 系统加载机制

| 时机 | 行为 |
|------|------|
| **流水线模式 - 章节开始** | 自动加载最近3章锚点（优先级高于 summary_index.md）|
| **互动模式 - INIT** | 自动加载最近3章锚点 |
| **互动模式 - 重锚周期** | 每5回合重新加载最近3章锚点 |
| **手动触发** | 用户输入 `[重新加载上下文]` |

### 22.3 三级概括压缩（summary_index）

`summary_index.md` 使用三级压缩策略以控制加载体积：

| 章节范围 | 压缩级别 | 呈现详度 |
|---------|---------|---------|
| 最近3章 | **完整** | 原始摘要全文（每章约150字）|
| 第4-10章以前 | **50%压缩** | 只保留关键转折和悬念 |
| 第10章以前 | **极简** | 每章仅1句话的极致摘要 |

> 此机制由 summary sub-agent 自动维护，写作时无需手动干预。

## 23. v8.1 新功能详解

> v8.1 对标 webnovel-writer 思路，在不引入外部依赖的前提下，落地了 9 项具体改进，提升爽点密度控制精度和节奏管理智能化程度。

### 23.1 5型钩子分类 + 强度分级（P1.1）

**功能点**：章末钩子不再只是「有没有」，现在分 5 种类型并追踪历史，防止连续 3 章同类型导致读者审美疲劳。

| 钩子类型 | 英文 ID | 说明 |
|----------|---------|------|
| 危机未解 | `crisis` | 主角/盟友陷入危险绝境，下章必须应对 |
| 奖励前置 | `reward_pre` | 大奖励即将到手但尚未落袋，点击欲极强 |
| 反转预告 | `reversal` | 已知信息被颠覆，读者急需知道真相 |
| 新威胁登场 | `new_threat` | 强敌/阴谋浮现，压迫感骤升 |
| 系统触发 | `system_call` | 系统弹窗/任务激活/升级临界（系统文专属）|

**强度分级**：`strong`（qd_07 ≥ 8）/ `medium`（6-7）/ `weak`（< 6，触发警告）

**锚点 frontmatter**：每章锚点文件新增 YAML 元数据头，包含 `hook_type`、`hook_strength`、`opening_pattern`、`emotion_rhythm`、`ending_time`、`ending_emotion`。

**新增 self_check 规则**：
- `hook_type_tracking`：验证锚点文件已写入 hook_type/hook_strength 字段
- `hook_type_variety`：连续 3 章同类型钩子触发告警（`overridable=true`）

### 23.2 8型爽点扩展（P1.2）

**新增两种套路**（写入 `methodology.yaml` law_5）：

**迪化误解**：主角随意行为 → 配角脑补升华 → 读者优越感暴击
- 核心：读者知道真相而配角不知的信息差
- 配角反应越煞有介事，爽感越强

**身份掉马**：长期伪装 → 关键时刻揭露 → 周围震惊崩塌
- 公式：身份差距 × 伪装期长度 × 揭露场合公开度 = 爽感强度
- 必须配合三层震惊链（执行者→亲历者→权威）

### 23.3 7型微兑现 + per-genre 基线（P1.3）

**7种微兑现类型**（写入 `self_check_rules.yaml`，rule_id: `micro_payoff_density`）：

| # | 类型 | 示例 |
|---|------|------|
| 1 | 信息兑现 | 悬念揭开、身份确认 |
| 2 | 地位兑现 | 主角地位可见提升 |
| 3 | 资源兑现 | 获得装备/钱财/盟友 |
| 4 | 反制兑现 | 成功反制反派压迫 |
| 5 | 情感兑现 | 情感期待得到回应 |
| 6 | 认知兑现 | 悬念阶段性解释 |
| 7 | 技能兑现 | 金手指以新方式展示 |

最小密度：爽文/言情每章 ≥ 1 次，虐恋/正剧每章 ≥ 1 次（质量 > 数量）。

### 23.4 压扬比例 per-genre（P1.4）

**新增 `genre_tags.yaml` — `tension_release_ratios` 节**（每 10 章分配基线）：

| 题材 | 压迫:兑现 | 最大连续压迫章数 |
|------|-----------|------------------|
| 爽文 | 3:7 | 2 章 |
| 正剧 | 5:5 | 3 章 |
| 虐恋 | 7:3 | 5 章 |
| 悬疑 | 4:6 | 3 章 |
| 言情 | 4:6 | 3 章 |

### 23.5 Override Contract 软规则豁免（P1.5）

**问题**：某些 WARN 规则在特定场景（刻意铺垫章/题材约束/用户明确指令）下应该被允许豁免。

**实现**：
- `self_check_rules.yaml` 中 `overridable=true` 的软规则，当触发 WARN 时向用户展示豁免选项
- 用户选择豁免后，自动写入 `state.json → override_contracts` 数组，包含 `rule_id`、`chapter_number`、`rationale_type`、`rationale_note`、`overridden_at`
- 未记录的豁免在下章执行时视为未曾豁免，重新触发检查

**豁免理由类型**：`user_explicit_intent`（用户明确指令）/ `genre_constraint`（题材约束）/ `deliberate_tension_chapter`（刻意铺垫章）/ `deliberate_focus_arc`（刻意聚焦弧光）

**目前支持豁免的规则**：`hook_type_variety`、`micro_payoff_density`、`strand_balance_check`

### 23.6 Strand Weave 三轨节奏系统（P2.6）

**问题**：单一追踪「爽点有没有」不够，需要追踪宏观叙事节奏的三条轨道平衡。

**三轨定义**：

| 轨道 | 名称 | 典型内容 |
|------|------|----------|
| Quest | 主线推进 | 金手指发动/实力突破/核心目标推进 |
| Fire | 危机激化 | 新敌登场/阴谋揭露/压迫值骤升 |
| Constellation | 世界观拓展 | 设定揭示/新势力出现/人物背景深化 |

**平衡规则**（写入 `self_check_rules.yaml`，rule_id: `strand_balance_check`）：
- Quest 连续 ≥ 5 章无其他轨 → 告警单调
- Fire 间隔 > 10 章 → 告警失去张力  
- Constellation 间隔 > 15 章 → 告警世界观停滞

**追踪机制**：`state.json → strand_tracker`（新增字段）；每章完成后，从 `scene_plan.dominant_strand` 读取并更新。

**场景规划标注**：`scenes/prompt.md` 场景清单和 scene_plan 文件新增 `主轨` 列和 `dominant_strand` 字段。

### 23.7 OOC 人设一致性检查（P2.7）

**位置**：写后质量评估阶段，八维度评分之后额外执行。

**机制**：
1. 从 kb_slice 读取本章涉及角色的 `personality` 字段
2. 检查正文行为是否与人设关键词存在断层
3. OOC 信号：行为断层 / 决策动机矛盾 / 说话方式与身份不符
4. 单处告警（warn），2处及以上打回 sub-agent 重写（halt）

> 依赖 KB 中角色 `personality` 字段的完善程度；字段缺失时跳过检查并发出 warn。

### 23.8 伏笔紧迫度排序（P2.8）

**位置**：Pre-write 检查 1（读取场景规划）完成后立即执行。

**机制**：
- 读取 `kb_slice.foreshadowing.planted` 中所有 `status=pending` 的伏笔
- 计算 `remaining = target_chapter - current_chapter`
- 三级标注：🔴 ≤5章（必须处理）/ 🟡 6-10章（建议处理）/ ⚪ >10章（监控中）
- 🔴 级伏笔自动注入 `dynamic_commands`，成为写作硬约束

> 需在 KB 的 foreshadowing.planted 条目中填写 `target_chapter` 字段才能生效排序；未填写视为 remaining=999。

### 23.9 结构化 chapter_meta（P2.9）

**功能**：将每章锚点的结构化元数据持久化到 `state.json`，而不只是存在 `.md` 文件里，让后续章节可以程序化查询历史钩子分布。

**写入时机**：步骤 8（更新 state.json）时，读取 anchor 文件 frontmatter 并写入 `state.json → chapter_meta["{N}"]`。

**字段**：`hook_type`、`hook_strength`、`opening_pattern`、`emotion_rhythm`、`ending_time`、`ending_emotion`。

**升级路径**：已有项目执行 `init.py --upgrade` 后自动补充 `chapter_meta`、`strand_tracker`、`override_contracts` 三个新字段（patch 9/10/11）。

4. 无情节支撑的实体：WARNING 提示遗漏
5. 输出 token：`[KB_HEAL: 自愈=N条 | 警告=N条]`

## 28. 全局叙事DNA系统（v8.5）

v8.5 引入**全局叙事DNA**——一套跨项目持久化的作者风格基底层，解决"每个项目都要重新调教风格"的问题。

### 28.1 三层优先级说明

```
cycle_quirks.md（尾部注入，本卷临时，优先级最高）
  ↓
dna_human_*.yaml（项目级DNA，由 benchmark 模块提取）
  ↓
clone_*.yaml（风格克隆模块）
  ↓
style_modules/（题材风格模块）
  ↓
global_author_dna.yaml（跨项目基底层，优先级最低）
  ↓
内置规则（writing_rules.yaml 等）
```

| 层级 | 文件 | 作用范围 | 触发方式 |
|------|------|---------|---------|
| 基底层 | `~/.narrativespace/global_author_dna.yaml` | 所有项目 | 自动加载 |
| 项目层 | `.xushikj/config/dna_human_*.yaml` | 当前项目 | benchmark 模块自动生成 |
| 本卷层 | `.xushikj/style_logs/cycle_quirks.md` | 当前卷 | `/style cycle` 触发 |

### 28.2 环境配置

**首次初始化全局 DNA：**

```bash
# 在项目目录下运行（自动创建目录和副本）
python scripts/init.py --project-dir . --upgrade
```

这会自动：
- 创建 `~/.narrativespace/global_author_dna.yaml`（若不存在）
- 在项目 `.xushikj/config/` 创建本地副本
- 创建 `style_actions/`、`style_logs/` 目录

**自定义全局 DNA 路径：**

```bash
python scripts/init.py --project-dir . --global-dna-path /path/to/my_dna.yaml
```

或设置环境变量：

```bash
export NARRATIVESPACE_GLOBAL_PATH=/path/to/.narrativespace/
```

### 28.3 `/style` 命令用法

在任意写作/互动模式对话中触发风格调整：

```
/style 本章对话太正式了，需要更口语化
/style 主角独白太长，需要更短平快
/style cycle 这卷风格偏轻快，本卷临时调整为更沉重的基调
```

**系统自动执行：**
1. 解析风格描述 → 生成结构化 JSON
2. 保存到 `.xushikj/style_actions/pending_001.json`
3. 提示运行 `update_style_rule.py` 落盘

**手动落盘：**

```bash
python scripts/update_style_rule.py --project-dir .
```

**作用域选择：**

| 作用域 | 指令 | 效果 |
|--------|------|------|
| `cycle` | `/style cycle 描述` | 写入 cycle_quirks.md（本卷结束归档） |
| `project` | `/style project 描述` | 写入 dna_human_custom.yaml（当前项目） |
| `global` | `/style global 描述` | 写入全局 global_author_dna.yaml（所有项目生效） |

默认作用域：`project`（当前项目）。

### 28.4 DNA 提纯（consolidate_dna.py）

每卷结束时运行 DNA 提纯，清理低质量规则：

```bash
# 随卷末快照一起触发
python scripts/volume_snapshot.py --project-dir . --volume 3 --consolidate-dna

# 单独运行提纯
python scripts/consolidate_dna.py --project-dir . --current-volume 3

# 仅分析不修改（dry-run）
python scripts/consolidate_dna.py --project-dir . --current-volume 3 --dry-run
```

**六阶段提纯流程：**

| 阶段 | 操作 |
|------|------|
| A | 加载全局 DNA |
| B | 语义去重（Ollama 余弦相似度 > 0.85，降级到精确匹配） |
| C | 冲突检测（极性分析，标记人工确认，不自动处理） |
| D | 权重衰减：gap > 3卷未触发 → weight -20%；weight < 30 → 归档 |
| E | Token预算强制：超出50条按 weight×recency 排序归档 |
| F | 写回全局文件 + 本地副本 + 更新 `archived_dna.yaml` |

### 28.5 规则恢复

归档规则保存在 `.xushikj/config/archived_dna.yaml`，可手动查看和恢复：

1. 打开 `archived_dna.yaml`，找到目标规则
2. 将规则条目（去掉 `archived_at`/`archived_reason` 字段）复制回 `global_author_dna.yaml` 对应类别
3. 重置 `weight: 100` 和 `last_triggered_vol`

### 28.6 注意事项

- **冲突规则需人工处理**：consolidate_dna.py 不自动删除语义冲突规则，仅输出报告
- **全局规则优先级最低**：项目级 dna_human_*.yaml 中相同 category+id 的规则会自动覆盖全局规则
- **cycle_quirks 卷末自动归档**：由 volume_snapshot.py 负责清空本卷临时规则并归档到 `## 归档记录` 区域

## 29. 对话执行边界与检查点管理（v8.5 新增）

本章节说明如何合理规划对话边界，以避免 Token 耗尽导致的上下文丢失问题。

### 29.1 核心原则：一个模式，一个对话

**强烈建议**：Mode A（流水线）和 Mode B（交互式 TRPG）**必须分开在不同对话中运行**，不要在同一次对话内连续完成两种模式。

**原因**：
- 完整执行一种模式（Step 0-11）约消耗 60-80% 的 Token 预算
- 在同一对话中执行两种模式必然触发自动摘要，造成执行状态信息丢失
- 摘要恢复后精度有损失，可能导致角色细节、伏笔记录不一致

### 29.2 每步完成后：保存检查点

每执行完一个重要步骤后，建议手动在项目目录创建或更新 `session_checkpoint.md`：

```markdown
# 对话检查点

- 当前步骤：Step N 完成
- 最后创建的文件：.xushikj/outline/xxx.md
- 已写章节数：N
- KB 实体数：角色 N / 地点 N / 物品 N
- 伏笔状态：planted N 条，resolved N 条
- 当前 cycle：cycle_id=N, cycle_status=XXX
- 下一步：Step N+1
- 备注：（记录任何需要新对话继续时的重要上下文）
```

**触发时机**：
- Step 0、Step 3、Step 6、Step 7 完成后（规划阶段关键节点）
- 每章写作完成后
- 每次 Humanizer 处理完成后
- 任何将要切换到新对话之前

### 29.3 Token 耗尽后的恢复流程

若对话 Token 耗尽（系统自动摘要），按以下步骤在新对话中恢复：

1. 打开之前保存的 `session_checkpoint.md`
2. 在新对话中粘贴检查点内容（约 200-500 字）
3. 触发词：`叙事空间` 或相应模块触发词
4. 告知系统当前步骤和需要继续的操作
5. 如有重要状态文件（`state.json`、`knowledge_base.json`），提示系统读取

**注意**：摘要机制只能保留核心内容，细节（如伏笔的具体措辞、角色的感官特征）可能丢失。这是技术限制，通过勤保存检查点可以最大程度减少损失。

### 29.4 最佳实践对照表

| 场景 | 推荐做法 |
|---|---|
| 新项目启动 | 一个对话完成 Step 0-6（规划阶段）|
| 知识库初始化 | 使用 `scripts/kb_init_example.py` 脚本，不在对话中粘贴大段 JSON |
| 正式写作 | 每 3-5 章换一个新对话（视对话 Token 消耗速度调整）|
| Mode A + Mode B | 两个完全独立的对话，不混用 |
| 长期项目（50+ 章）| 每 10 章做一次 `volume_snapshot.py` 整理，清理 KB 并归档 DNA |

### 29.5 KB 写入推荐方式

**不推荐**：在对话中直接粘贴 JSON 或通过 `python -c` 执行含中文字符的 JSON 字符串（有引号转义风险）。

**推荐**：使用 `scripts/kb_init_example.py`：

```bash
# 1. 编辑 scripts/kb_init_example.py 中的 kb 字典
# 2. 运行：
python scripts/kb_init_example.py
# 3. 确认输出：.xushikj/knowledge_base.json
```

此脚本为纯 Python 文件，中文字段完全安全，无需任何转义处理。
