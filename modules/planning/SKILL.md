# 叙事空间创作系统 - 规划模块 (opencode)

`opencode-xushikj-guihua` 负责**分段规划**，目标不是一次性写死全书，而是为“当前创作循环”输出可直接开写的规划包。

## 分段规划目标

本模块每次只规划当前 cycle 所需内容，并在满足写作门槛后立即交接给写作模块：

- 不做全书刚性前置规划
- 只做当前循环的“最小可写规划包”
- 与 `opencode-xushikj-chuangzuo` 路由契约保持一致

但“分段规划”不等于“跳过访谈”。规划模块必须先充分了解用户意图，再开始生成；每一步生成完成后，必须等待用户确认，不能自动越步。

## 输入上下文

执行前读取 `.xushikj/state.json`：

- `rolling_context.cycle_id`
- `rolling_context.cycle_size`
- `rolling_context.cycle_status`
- `rolling_context.granularity`
- `rolling_context.pending_granularity`

## 粒度驱动的分段规划规则

| `granularity` | 规划范围 | 规划深度基线 | 默认交付 |
|---|---|---|---|
| `single` | 下一章（1章） | 高聚焦，节奏快 | 1章 mini outline + Layer-1 卡 |
| `3-5` | 下一循环（3~5章） | 平衡效率与连贯性 | 3~5章 mini outline + Layer-1 卡 |
| `volume` | 下一卷块（20~50章） | 宏观先行，章级可分批细化 | volume chunk 纲要 + 当前轮 Layer-1 卡 |

> 注意：`volume` 粒度也不代表必须完整细化所有章节场景；仅当前轮进入写作的章节必须具备 Layer-1。

## 步骤映射与产出路径

| 步骤 | 名称 | 产出文件 | 是否强制 |
|------|------|----------|---------|
| 1 | 一句话概括 | `.xushikj/outline/one_sentence.md` | 是 |
| 2 | 一段式概括 | `.xushikj/outline/one_paragraph.md` | 是 |
| 2.5 | 世界观与力量体系设定 | `.xushikj/outline/worldview_and_system.md` | **条件触发** |
| 3 | 人物卡片 | `.xushikj/outline/characters/` 目录，每角色独立文件 | 是 |
| 4 | 一页大纲 | `.xushikj/outline/volume_{V}_one_page.md` | 是 |
| 5 | 人物弧光大纲 | `.xushikj/outline/character_arcs.md` | 可选（步骤4确认后须主动询问用户意愿） |
| 6 | 四页大纲 | `.xushikj/outline/volume_{V}_four_pages.md` | 可选（步骤5完成/跳过后须主动询问用户意愿） |
| 11 | 书名与简介 | `.xushikj/outline/title_and_synopsis.md` | **上架/发布前必须** |

### 人物卡片目录规范（步骤3）

每个角色必须单独存为一个文件，路径格式：`.xushikj/outline/characters/{char_id}.md`。

最小字段集（每张卡片必须包含）：

1. `名字`（含外号/绰号）
2. `角色类型`（主角 / 女主 / 反派 / 配角 / 灰色）
3. `价值观`
4. `抱负`
5. `当前目标`
6. `内在矛盾`（内在冲突）
7. `行为底层逻辑`（不妥协的执念）
8. `压力反应基线`（高压下的惯性行为）
9. `欲望-恐惧-羞耻-债务` 最小字段

禁止将所有角色写入同一文件，必须按文件夹管理。已存在 `outline/characters.md` 的旧项目，首次规划时必须拆分为独立文件。

### 步骤2.5 条件触发规则（世界观与力量体系）

步骤2.5 不是无条件强制步骤，而是**条件触发步骤**。命中以下任一条件时，必须补全后才能交接写作：

1. `target_platform=fanqie` 时：世界观背景 + 系统/金手指规则必须完整
2. 含异能/修炼/系统/科幻设定的类型：力量边界与升级规则必须明确
3. 现实题材：核心信息差与权力结构必须说明

若步骤 2.5 被触发，产出文件必须包含：
- 世界背景（时代、地理、社会结构）
- 力量体系或金手指规则（触发条件、限制、升级路径）
- 主角起点与当前阶段

若为纯现实、日常、无超自然规则的题材，且用户明确表示不需要独立世界观/力量体系文件，则允许跳过独立文件，只在一段式概括与人物/卷纲中保留必要背景约束。

## 当前循环交付物（必须）

每轮规划完成时，至少产出：

### pipeline 模式

1. **当前循环章节大纲**
   - 当前 cycle 的章节目标和关键推进节点
2. **Layer-1 场景卡清单**
   - 当前 cycle 每章至少 1 条"目标/冲突"场景卡
3. **延续备注**
   - `next_cycle_hint`，用于下一轮承接与钩子延续

### interactive 模式

1. **当前循环核心规划包**
   - 至少包含 Core Meta、角色基线、必要背景约束
2. **用户可直接给方向的写作入口**
   - 当前章或当前轮可接受即时指令，不要求预写 Layer-1
3. **延续备注**
   - `next_cycle_hint`，用于下一轮承接与钩子延续

## 用户访谈与逐步确认（硬规则）

规划阶段必须采用“先问后写、一步一确认”的交互方式。

### 访谈先于生成

在开始步骤 1 之前，至少确认以下内容：

1. 作品题材与类型混搭边界
2. 目标平台或默认平台
3. 核心卖点 / 一句话 premise
4. 主角基本身份、核心欲望与主要冲突
5. 是否包含世界观、力量体系、金手指、悬疑机关、感情主线等关键要素
6. 用户明确禁区与偏好（如不要后宫、不要系统、节奏偏快/偏慢）

若上述信息缺失：

1. 必须先向用户提问
2. 禁止直接生成世界观、力量体系、角色卡、四页大纲
3. 禁止用“默认补全”替代必要访谈

仅当用户明确说“这部分交给你设计”时，才允许代拟；但代拟结果仍必须请求用户确认。

### 单步确认后才能进入下一步

步骤 1、2、2.5、3、4、5、6、11 全部遵守以下规则：

1. 先询问当前步骤的具体需求、偏好、禁区和想保留的设定
2. 仅在当前步骤信息足够后，才输出该步骤草案
3. 明确询问用户是“确认 / 修改 / 补充”
4. 只有在用户确认后，才允许进入下一步骤
5. 若用户要求修改，继续停留在当前步骤修订，不得自动推进
> ⛔ **HARD STOP**：以上规则为系统级行为约束，任何效率理由或"用户可能已理解"的推断均不得豁免。步骤间的自动推进属于系统缺陷，而非贴心服务。步骤5、6为可选步骤，必须在用户明确选择后才能进入，沉默不等于授权。
### 每一步的最小询问要求

为防止“开头问一次，后面全自动跑完”，每一步生成前至少补做一次针对性询问：

1. 步骤 1：询问题材、核心卖点、主角处境、目标读者印象
2. 步骤 2：询问故事主线、核心冲突、阶段目标、预期爽点/情绪走向
3. 步骤 2.5：询问世界观底色、规则类型、力量来源、代价、边界、升级方式
4. 步骤 3：询问角色定位、关系结构、性格底色、执念、雷点、感情线安排
5. 步骤 4：询问故事展开节奏、关键转折、阶段高潮、是否保留某些桥段
6. 步骤 5：询问角色弧光方向、成长代价、关系变化、是否需要反转或黑化
7. 步骤 6：询问长线伏笔、卷级结构、阶段回收点、后续升级预期
8. 步骤 11：询问平台风格、书名偏好、简介强调点、是否避开某类词汇

若用户回答“你先提案”，也只表示该步骤允许代拟，不表示后续步骤一并放权。进入下一步前仍要重新询问。

### 特殊强调：步骤 2.5 不得抢跑

世界观与力量体系是高影响设定，命中以下任一情况时，必须先问清楚再写：

1. 用户尚未说明题材底色
2. 用户尚未说明是否需要系统、异能、修炼、规则怪谈、超自然机制
3. 主角成长路径、资源来源、代价机制不明确

在这些信息未补齐前，不得直接生成 `worldview_and_system.md`。

## 番茄里程碑节点（新增）

当项目目标平台为番茄（`target_platform=fanqie`）时，规划阶段必须显式标注三个字数里程碑：

1. 8万字：验证期入口（必须安排中强度转折或势能抬升）
2. 10万字：首秀窗口（必须安排强钩子或关键反杀节点）
3. 20万字：书测窗口（必须安排阶段性成果或主线揭示）

执行要求：

1. 在当前 cycle mini outline 中标注“本轮涉及的里程碑章节号（如有）”
2. 命中里程碑章节时，必须在场景层触发 Layer-2 深化
3. 若当前 cycle 未命中里程碑，需在 `next_cycle_hint` 中写明最近里程碑与预备动作

## 写作交接条件（硬规则）

当下列条件满足时，规划模块必须停止继续扩展并交接：

- 最小可写门槛（Minimum Writable Threshold）= PASS
- 本轮进入写作所需的规划步骤均已获得用户确认

追加条件：

1. `writing_mode=pipeline`：当前 cycle 的 Layer-1 场景卡完整
2. `writing_mode=interactive`：Interactive Minimum Writable Threshold 满足，不要求预写 Layer-1

交接目标：

- 流水线模式 -> `opencode-xushikj-xiezuo`
- 互动模式 -> `opencode-xushikj-hudong`

## 需要加载的配置

| 配置文件 | 用途 | 必须 |
|----------|------|------|
| `.xushikj/config/methodology.yaml` | 八大法则，贯穿每一步的创作指导 | 是 |
| `.xushikj/config/meta_rules.yaml` | 输出语言与符号标准化 | 是 |

运行期配置必须来自 `.xushikj/config/`，不得回退读取 Skill 自带 `config/`。

## 变更处理：中途改目标

用户在当前循环中途修改目标时：

1. 仅回写和重排**当前 cycle**计划
2. 保留已完成章节与既有状态
3. 不触发“全书重规划”

## 执行流程（简版）

1. 读取状态 + 粒度
2. 先执行用户访谈，补齐最低规划输入；信息不足时只提问，不生成设定
3. 步骤1：先问一句话概括的具体诉求 → 生成草案 → 请求用户确认 → 确认后写入 `.xushikj/outline/one_sentence.md`
4. 步骤2：先问一段式概括的具体诉求 → 生成草案 → 请求用户确认 → 确认后写入 `.xushikj/outline/one_paragraph.md`
5. **步骤2.5（条件触发）**：命中触发条件时，先问世界观/力量体系的具体诉求 → 在关键信息明确后生成草案 → 请求用户确认 → 确认后写入 `.xushikj/outline/worldview_and_system.md`
6. 步骤3：先问人物设计的具体诉求 → 生成人物卡片草案 → 请求用户确认 → 确认后写入 `.xushikj/outline/characters/{char_id}.md`（每角色独立文件）
7. 步骤4-6：每一步都先询问该步具体需求，再生成、再确认，不得越步
8. 生成当前 cycle 章节大纲
9. 若 `writing_mode=pipeline`：补齐当前 cycle Layer-1 场景卡
10. 若 `writing_mode=interactive`：仅整理当前轮可直接开写的核心规划包，不强制预写 Layer-1
11. 写入/更新 `next_cycle_hint`
12. 执行门槛检查：
   - PASS 且相关步骤已确认 → 交接写作
   - FAIL 或存在未确认步骤 → 返回缺失清单/待确认项，仅补当前缺口

## 参考规则接入（新增）

本模块新增引用：`references/character-construction-rules.md`。

规划阶段执行要求：

1. 核心角色必须补齐“欲望-恐惧-羞耻-债务-矛盾”最小字段
2. 主角与核心配角需定义压力反应基线
3. 每轮规划至少标记一名核心角色的弧线阶段推进目标
4. 命中拒收条件时，先补角色约束再交接写作

## 示例

### 示例 A：默认 3-5 章交接写作
- 输入：`granularity=3-5`、`cycle_status=plan_light`
- 规划输出：步骤2.5世界观 + 第 1-4 章章节大纲 + 4 份 Layer-1 场景卡 + `next_cycle_hint`
- 门槛结果：PASS
- 路由：交接 `opencode-xushikj-xiezuo`（流水线）或 `opencode-xushikj-hudong`（互动）

### 示例 B：用户“先写再补”
- 输入：用户要求“先写第3章，后面再补细”
- 处理：仅确保第3章进入写作所需 Layer-1 完整，其他章维持分段待补
- 路由：当前章可写；后续在 `deepen/continue` 阶段补齐，不扩大为全书重规划

## 边界声明

- 本模块不直接写正文
- 本模块不要求 Layer-2 全量完成
- 本模块不覆盖旧项目已完成内容
