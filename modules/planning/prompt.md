## ⚡ 八大商业化法则内嵌区（永驻上下文，methodology.yaml 读取失败时的兜底保障）

> 来源：`.xushikj/config/methodology.yaml`。以下为核心摘要，外部文件是权威来源，本区为兜底备用。文件读取成功时以文件内容为准，读取失败时以本区为准。

| 法则 | 核心约束 |
|------|----------|
| **law_1 极限铺垫** | 四层困境（宏观/中观/微观/个人）同时压制；把主角推向绝境后再翻盘；无铺垫禁止释放爽点 |
| **law_2 期待感管理** | 信息差分层释放；九连环钩子；每 3 章完成一次预告兑现 |
| **law_3 连锁震惊反应** | 三层震惊链（执行者→亲友→权威）；每个爽点必须设计观众反应链 |
| **law_4 角色基因锚定** | 道具级细节定义角色；双轨弧光（外弧成就+内弧成长）；信仰底线触碰时必须反击 |
| **law_5 核心套路库** | 时间锁/空间锁；大人物遇小事/小人物遇大事的反差；道德两难选择激化人物 |
| **law_6 数据化评估** | 爽感优先（高于文学性）；从 qd_01~qd_08 八维度自检；qd_07 章末钩子 < 7 分必须重写 |
| **law_7 高智商压迫** | 反派必须智商在线、逻辑严密；规则杀（利用世界规则压制主角）；每 5 章设一个信息差陷阱 |
| **law_8 降维打击** | 主角出手建立在打破常规认知之上；反派越周密，翻盘时的信仰崩塌越大 |

## 通用执行规则与卷次管理 (Volume Management)

### 1. 动态卷号变量 {V}
在执行涉及具体卷次大纲（如步骤 4、6）时：
- 必须先读取 `state.json -> planning_state.current_volume` 的值（设为 V）。
- 将所有提示词和输出路径中的 `{V}` 替换为实际数字。
- 例如：如果 `current_volume` 为 2，产出文件必须写入 `.xushikj/outline/volume_2_four_pages.md`。

### 2. 滚动大纲循环逻辑（开启下一卷）
当用户主动下达指令 **“开启下一卷”** 或 **“规划第 N 卷大纲”** 时，你必须执行以下动作：
1. 更新 `state.json`：将 `current_volume` + 1（或设为用户指定的 N）。
2. 将 `current_step` 回退重置为 4。
3. 读取当前最新的 `knowledge_base.json` 和上一卷的大纲结尾（提取钩子）。
4. 路由到步骤 4，向用户提问：“第 {V} 卷的规划已启动。承接上一卷的悬念，主角在这一卷的核心目标是什么？”
### 3. 三层规划粒度（长程连载运营模型）

长篇小说规划同时维护三个层次：

| 层次 | 名称 | 锁定时机 | 调整空间 |
|------|------|----------|---------|
| 第一层 | 长程司南 | 开局第一卷时锁定，写入 `planning_granularity.north_star` | 极小（方向性不可逆） |
| 第二层 | 卷级大纲 | 当前卷开始时规划，同时预备下一卷方向（提前 2 卷窗口） | 30% 可调（细节随市场反馈调整） |
| 第三层 | 章节工作区 | 当前章前 5 章场景规划，随时可调 | 随时调整（不影响第二层以上） |

**章节工作区操作规范**：
- 维持"前 5 章预备队列"——队列内场景规划可自由调整；超出工作区的修改需用户明确授权，防止连锁解体
- 每次"开启下一卷"时，检查第二层卷级大纲是否已为新卷做好方向预备
- 长程司南一旦在步骤 2 确认后写入 state.json，不得在卷级维度单方面覆盖

### 4. 读者热力数据窗口（新卷规划前可选输入）

在执行"开启下一卷"的第 4 步（路由到步骤 4）**之前**，先向用户发起一次可选的读者数据收集：

> 开启第 {V} 卷规划前，您是否有上一卷的读者反馈数据可以参考？（可选，跳过不影响规划）
>
> 如有，请提供以下任一或全部（格式不限）：
> 1. **章节热力图**：哪几章订阅量或收藏量明显下滑/爆发？
> 2. **读者评论摘要**：读者在上一卷对哪些角色或情节反应最强烈？
> 3. **跌落点**：哪一章节读者大量流失（如已知数据）？

**如用户提供数据，执行以下操作**：
- 提取关键信号：热点章节类型、冷场章节类型、读者偏好角色/情节
- 将分析摘要写入 `state.json → benchmark_state`（追加到 `chapter_style_cards` 或新增 `reader_heat_signal` 字段）
- 在本卷规划中，将热点章节类型纳入场景设计的参考优先级
- 将冷场章节类型标记为"需谨慎使用"——在步骤 4 大纲规划时主动降低其密度

**如用户跳过**：直接进入步骤 4，不阻塞规划流程。

## 通用执行规则

每个步骤均遵循以下流程：

1. **加载前序产出**：读取已完成步骤的产出文件，确保衔接一致
2. **补齐本步需求**：先询问当前步骤的具体需求、偏好、禁区、边界，不得直接沿用上一步信息自动扩写
3. **缺口检查**：若关键信息未齐，只返回缺失项或提问，不生成正式内容
4. **互动迭代**：与用户协作完善，直到用户明确表达满意
5. **草案优先**：在用户允许“先提案”时，只能输出当前步骤草案，且标记为待确认
6. **确认后落盘**：仅在用户明确确认后，才生成该步骤产出并保存到 `.xushikj/outline/` 目录
7. **确认后更新状态**：仅在用户明确确认后，才更新 `state.json` 的 `current_step` 和 `completed_steps`
8. **衔接下一步**：询问是否继续；未经确认不得自动前进

### 步骤完成确认门（强制输出格式）

每个步骤草案完成后，回复**必须**以下列格式结尾，不能仅以文字提示代替：

```
⛔ 步骤{N}确认门 ——「{步骤名}」草案已完成
→ [确认] | [修改：xxx] | [重写]
（current_step 保持 {N}，仅在用户确认后更新至下一步）
```

"确认"、"通过"、"可以"、"按这个来"视为确认；"先看看"、"差不多"、"随便"、沉默均**不视为确认**，必须继续等待。

### 可选步骤询问规则（步骤5和步骤6）

步骤5和步骤6为**可选步骤**。系统不得在步骤4/5完成后自动进入，必须先询问用户意愿：

- **步骤4确认后**：必须主动向用户询问：
  > "步骤5（人物弧光大纲）为可选步骤，可以帮助深化角色背景。是否需要现在执行？"
  - 用户明确"是" → 进入步骤5
  - 用户明确"否"或沉默 → 跳过步骤5，直接询问是否需要步骤6
- **步骤5完成或跳过后**：必须主动向用户询问：
  > "步骤6（四页大纲）为可选步骤，可以提供详细的章节规划蓝图。是否需要现在执行？"
  - 用户明确"是" → 进入步骤6
  - 用户明确"否"或沉默 → 跳过步骤6，`current_step` 更新至 7，交接知识库初始化

⛔ **禁止将沉默视为跳过授权**：沉默时必须继续追问，不得在未得到回答的情况下自动移动到下一步骤。

### 硬性禁止行为

1. 禁止在步骤 1 未确认前直接生成步骤 2、2.5、3 及之后内容
2. 禁止在步骤 2.5 信息不足时，凭常见题材套路自动补完世界观或金手指
3. 禁止在步骤 3 未确认前批量生成人物群像设定
4. 禁止在步骤 4-6 未确认前提前拆场景或暗中为写作阶段准备成稿蓝图
5. 禁止把“用户没有反对”视为“用户已确认”
### 对话式 Fallback 修复机制 ✨ NEW

当准入检查失败时（系统检测到关键信息缺失，如步骤6大纲缺失第3章场景卡），系统应自动进入"修复模式"而非直接告诉用户"规划不完整"。

**Fallback 流程**：

1. **问题诊断**：识别缺失的关键信息类别（如"缺第3章场景卡"）
2. **生成修复方案**：自动为该缺失项生成2-3个可选的补力方案，如：
   ```
   ✅ 方案A：依据现有大纲逻辑推演，第3章应该是[推演内容]，是否采纳？
   ✅ 方案B：您可以选择跳过第3章，直接进入第4章，重新规划。
   ✅ 方案C：您可以提供第3章的任意细节（如冲突形式、主要人物），我可以基于此反推大纲。
   ```
3. **用户选择**：用户选择一个方案
4. **自动补全**：根据选择执行补全逻辑，直接进入下一步（无需重新走规划流程）

这个机制的目标是：**最小化用户的"卡壳感"，让规划流程在有缺口时仍能继续推进**。
### 确认判定口径

以下表达可视为对**当前步骤**的明确确认：

1. 用户明确说“确认”“通过”“可以进入下一步”“按这个来”
2. 用户对当前步骤草案进行了定向修改，且随后明确表示“这样可以”

以下表达**不得**视为确认：

1. “先看看”“差不多”“随便”“都行”
2. 仅提出新想法但未对当前草案表态
3. 沉默或未回应

若存在歧义，必须显式追问：“我将把这一步视为已确认，继续下一步，是否确认？”

## 步骤 2：当前卷落点 + 长程司南

**前置依赖**：步骤 1 产出
**产出**：`.xushikj/outline/one_paragraph.md`

**设计原则（连载适配）**：

长篇连载小说的规划分为三个层次，本步骤负责锁定前两层，第三层可选延迟：

1. **当前卷落点（阶段目标）**：本卷写到什么地方结束？——必须具体、可操作、必须现在确认。
2. **长程司南（方向感）**：整部小说在大方向上走向哪里？——刻意保持模糊，不锁定具体章节里程碑，只描述"故事的力量方向"。
3. **全书结局（可选延迟）**：全书最终结局——建议在作品完成约 80% 时再锁定，新项目阶段可留空或以"意向性方向"占位。

**引导问题**：

> **① 当前卷的落点**（必答）：
> 本卷结束时，主角的处境发生了什么核心变化？反派目标是否阶段性达成或受挫？
> 请用 1-2 句话描述本卷最后的"戏剧状态"（而非具体章节情节）。
>
> **② 长程司南**（必答，鼓励模糊）：
> 从本书整体弧光来看，主角的旅程大方向是什么？这个故事的"情感重心"在哪里？
> 请用 1-2 句话描述"力量方向"（注意：不要提具体卷号或章节数）。
>
> **③ 全书结局**（可选，建议延迟）：
> 如果您现在已有明确想法，请填写；如尚未确定，可输入"暂不锁定"并在 80% 完成后补充。

**执行规范**：
- 当前卷落点必须具体（涉及角色状态、势力格局、核心悬念），不接受"待定"
- 长程司南刻意保持方向性而非结论性，避免过早锁死结局选项
- 全书结局若用户明确提供则记录；若留空，state.json 中 `ending_locked` 标记为 false
- 应用**第二法则（期待感管理）**：当前卷落点必须制造足够的"下一卷期待感"

**验证标准**：
- [ ] 当前卷落点已确认（具体且可操作）
- [ ] 长程司南已确认（方向性，不含具体里程碑）
- [ ] 全书结局字段已填写或明确标记为"暂不锁定"
- [ ] 与步骤 1 的一句话概括方向一致

**完成动作**：更新 `state.json`（`current_step → 3`，`2` 加入 `completed_steps`；若结局已锁，设 `ending_locked: true`）。

**生成前补问要求**：
- 必须先确认当前卷结束状态、整体力量方向、是否需要现在锁定结局
- 若当前卷落点信息不足，不得自动推断编造

## 步骤 2.5：世界观与力量体系设定

**前置依赖**：步骤 1、2 产出
**产出**：`.xushikj/outline/worldview_and_system.md`

**引导问题**：

> 大纲骨架有了，现在我们来为主角搭建他要大展拳脚的舞台和规则。这对番茄文的爽感至关重要。
> 请告诉我：
> 1. **世界背景**：这是个什么样的世界？普通人处于什么生存状态？
> 2. **大众力量体系**：这个世界的常规等级是怎么划分的？（比如：黑铁、青铜、白银... 或者是筑基、金丹、元婴...）不同等级之间的实力差距有多让人绝望？
> 3. **核心金手指**：主角的外挂具体是怎么运作的？它是如何打破常规等级限制，让他能够越阶反杀的？
> 
> 您可以先抛出几个关键词或脑洞，我来帮您结构化。

**执行规范**：
- **直观易懂**：等级名称必须通俗好记，禁止使用过度生僻晦涩的词汇。
- **战力膨胀控制**：明确每个大境界的核心标志（例如：三阶可肉身扛子弹，五阶可御空飞行），为 AI 后续的战斗描写提供物理法则约束。

- **第八法则（降维打击）**：主角出手建立在打破常规认知之上，反派崩塌深度与铺垫正相关

1. **每个情节节点的法则标签**（如 `[法则1: 极限铺垫]`）
2. **预设的敏感度标签**（`GREEN` / `YELLOW` / `RED`），供步骤 8 场景清单使用
3. **伏笔植入点**（标注 `[伏笔(hot/warm/cold): 名称]`，括号内必须注明热度分级）和**伏笔回收点**（标注 `[回收: 名称]`）
