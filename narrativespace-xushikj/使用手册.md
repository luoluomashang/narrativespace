## 1. 系统简介

### 1.1 定位与目标

**叙事空间创作系统 v7.0**（Narrative Space Creation System）是一套面向番茄小说网男频商业网文的全流程自动化创作工具包。系统以 Claude Code Skill 形式交付，提供从作品立意到章节输出的完整闭环。

- **目标**：高质量、高爽感、商业化男频网文的批量生产
- **平台**：番茄小说网（番茄文风格，短句＋高信息密度）
- **技术栈**：Claude Code + MCP 工具（websearch、sequential-thinking）

### 1.2 核心能力

| 能力 | 描述 |
|------|------|
| 全流程规划 | 一句话创意 → 完整四页大纲，经过 12 步骤的系统性迭代 |
| 双模式写作 | 流水线模式（批量自动）+ 互动跑团模式（实时驾驶）|
| 动态知识库 | 全自动实体追踪、伏笔管理、角色弧光记录 |
| 八大创作法则 | 商业验证的剧情设计原则，内嵌于每次写作调用 |
| 质量双保险 | 8 维度评分 + Critic sub-agent 质检，确保章节合格率 |
| 去AI痕迹 | 专用 humanizer 后处理模块，消除 AI 文字痕迹 |

### 1.3 系统架构总览

```
叙事空间创作系统 v7.0
│
├── 规划引擎 (modules/planning)
│   ├── Step 0: 对标作品学习
│   ├── Step 1-2: 一句话 + 司南落点
│   ├── Step 2.5: 世界观与力量体系
│   ├── Step 3-6: 一页大纲 → 四页大纲
│   └── Step 11: 书名与简介
│
├── 知识库引擎 (modules/knowledge-base)
│   └── Step 7: KB 初始化 + 增量更新
│
├── 场景引擎 (modules/scenes)
│   └── Steps 8-9: 场景清单 + 场景规划
│
├── 写作引擎
│   ├── 流水线模式 (modules/writing)   — Step 10A
│   └── 互动跑团模式 (modules/interactive) — Step 10B
│
├── 对标引擎 (modules/benchmark)
│   └── Step 0 (可选): 风格分析
│

```

|------|------|------|

## 2. 快速开始

### 2.1 系统要求

- **Claude Code** CLI 或 VS Code Copilot Chat（支持 Skill 文件调用）
- **MCP 插件**（可选但推荐）：
  - `@modelcontextprotocol/server-brave-search` —— websearch 工具
  - `@modelcontextprotocol/server-sequential-thinking` —— sequential-thinking 工具

### 2.2 Skill 安装位置

本系统 SKILL.md 文件需放置在 Claude Code 能识别的 Skills 目录中：

```
~/.claude/skills/narrativespace-xushikj/SKILL.md
```

或工作区根目录下的 `.claude/skills/` 中。

### 2.3 初始化新项目

#### 三种运行模式

| 模式 | 命令 | 适用场景 |
|------|------|----------|
| **默认模式** | `python init.py --project-dir .` | 首次初始化；跳过已存在的文件，不覆盖任何数据 |
| **升级模式** | `python init.py --project-dir . --upgrade` | 已有项目升级系统版本；自动更新 `config/` 和 `references/`，**保留** `state.json`、`knowledge_base.json`、`memory.md` 等用户数据 |
| **强制模式** | `python init.py --project-dir . --force` | 重置项目；覆盖所有文件（包括用户数据）——**慎用** |

`--force` 与 `--upgrade` 不可同时使用。

#### 缺失文件自动检测

每次运行时，脚本会在执行写入前**扫描项目中的缺失文件**，按三类列出：

```
[!] 发现 5 个缺失文件（本次运行将自动补充）:

  [用户数据] (1 个)
    - state.json

  [参考文件] (1 个)
    - references\few_shot_examples.md

  [配置文件] (3 个)
    - config\workflow.yaml
    - config\writing_rules.yaml
    - config\style_modules\fantasy.yaml

是否补充上述缺失文件？[Y/n]
```

- 输入 `Y`（或直接回车）：补充所有缺失文件后继续
- 输入 `n`：退出，不做任何修改
- `--force` 模式跳过询问，直接执行

#### 基本用法

```bash
# 进入你的小说项目目录
cd /path/to/your/novel-project

# 首次初始化
python /path/to/narrativespace-xushikj/scripts/init.py --project-dir .

# 升级已有项目（保留创作数据）
python /path/to/narrativespace-xushikj/scripts/init.py --project-dir . --upgrade
```

**init.py 执行后创建的目录结构**：

```

├── state.json           # 全局运行状态
├── knowledge_base.json  # 动态知识库

```


### 2.5 首次使用推荐流程

```
1. 初始化项目目录（init.py）
2. Step 0（可选）：提供1-3部对标作品，系统学习风格
3. Step 1：输入你的核心创意概念（一句话）
4. Step 2：确定当前卷落点 + 全书司南
5. Step 2.5：确定世界观与金手指体系
6. Step 3：生成初始人物设定
7. Step 4：生成当前卷一页大纲
8. Step 5：生成人物弧光规划
9. Step 6：生成四页大纲（带伏笔）
10. Step 7：初始化知识库
11. Steps 8-9：规划场景
12. Step 10A 或 10B：开始写作
```

### 2.6 用户设定表单（user_settings_template.md）

`templates/user_settings_template.md` 是项目的**配置入口**，用于在对话开始时向 AI 声明核心参数。将此文件复制到你的项目目录，按需填写后粘贴到对话中。

**主要配置项**：

| 配置节 | 内容 |
|--------|------|
| 世界观与角色基础设定 | 对标作品、类型标签、世界背景、主角困境、金手指 |
| 扮演角色信息 | 角色名称、身份、性格、外貌、与用户关系 |
| 格式与输出设置 | 回复长度（A/B/C/D）、互动选项开关、规则自检等 |
| 特殊规则/状态管理 | 自定义追踪状态（好感度、爽感值等） |

**回复长度档位**：

| 档位 | 说明 |
|------|------|
| A | 每次 3000+ 字 |
| B | 每次约 1500 字 |
| C | 每次约 800 字 |
| D（默认）| 自主决定，倾向长回复，特定描写 ≥300 字 |

## 4. 宏观十二步工作流

系统按照 Steps 0 → 11 的固定顺序推进。每一步都有明确的**前置条件**、**输入**、**处理**和**输出**。

### Step 0：对标作品学习（可选）

| 项目 | 内容 |
|------|------|
| **触发条件** | 用户主动提供对标作品名称 |
| **MCP 工具** | `websearch`（查询书评、数据、排行信息）|
| **精度档位** | quick（快速）/ standard（标准）/ deep（深度）|
| **输出** | `.xushikj/benchmark/benchmark_analysis.md` |

**三档精度说明**：

- **quick**：仅读前 15 章 + 最后 5 章（2分钟内完成）
- **standard**：分层采样（前、中、高潮、结尾各采 10 章）
- **deep**：全文采样 + 评论区情绪分析 + webearch 外部书评

**Step 0 产出内容**：
- 爽感类型识别（打脸型/装逼型/系统流/重生流等）
- 节奏模式提取（章节字数分布、信息密度、钩子频率）
- 高频套路抽取（前10 + 后10章最常见剧情结构）
- 风格参数写入 `style_profile.yaml`

---

### Step 1：一句话概括

| 项目 | 内容 |
|------|------|
| **目的** | 锁定核心故事基因，后续所有规划的北极星基准 |
| **格式要求** | ≤25词；必须包含：主角+情境+方式+结果 |
| **画面感要求** | 闭上眼能看见一个具体画面 |
| **输出** | `.xushikj/outline/one_sentence.md` |

**一句话格式模板**：

```
[主角身份+处境] + [面临的绝境/挑战] + [使用的方式/金手指] + [最终获得的结果/地位变化]
```

**示例**：

```
被门阀废弃的少爷，意外激活祖先记忆，凭万年棋局算尽天下人心，
最终执棋坐断七大门阀。
```

---

### Step 2：当前卷落点 + 长程司南

| 项目 | 内容 |
|------|------|
| **目的** | 建立双层目标坐标（当前卷收尾 + 全书方向）|
| **写入字段** | `state.json > planning_state > planning_granularity.north_star` |
| **输出** | `.xushikj/outline/north_star.md` |

**两层内容**：

1. **本卷具体落点（30%可调）**：当前卷结束时，主角具体达到什么状态？哪个反派被打倒？获得什么资源？
2. **全书长程司南（极小幅可调）**：全书最终的意境画面、主角的终极境界、核心主题的升华点。

**司南三要素**：
- **情感意境**：全书的最终情绪底色（不是情节结局）
- **能力顶点**：主角金手指在全书末段的成就
- **核心命题**：故事想让读者相信的一件事

---

### Step 2.5：世界观与力量金手指体系

| 项目 | 内容 |
|------|------|
| **目的** | 定义规则体系，确保金手指逻辑自洽 |
| **MCP 工具** | `websearch`（查询硬核数据背书：量子、生物、历史等）|
| **输出** | `.xushikj/outline/worldview_and_system.md` |

**必须覆盖的内容**：

```
1. 世界类型：现实向/架空/玄幻/科幻/系统流/重生流...
2. 力量体系：等级划分（至少5级）+ 突破条件
3. 金手指设计：
   - 名称与表现形式
   - 触发条件（觉醒机制）
   - 成长路线（前期→中期→后期强度变化）
   - 合理性解释（为什么主角能有）
   - 核心限制（边界在哪里，不能无脑全知全能）
4. 世界核心规则：至少3条真实约束主角的规则
```

### Step 3：一页人物介绍

| 项目 | 内容 |
|------|------|
| **目的** | 建立核心角色双轨弧光 |

## 5. 三层规划粒度系统

本系统使用三层嵌套的规划结构，平衡长期一致性与短期灵活性。

### 5.1 三层概述

| 层次 | 名称 | 调整幅度 | 锁定时机 |
|------|------|---------|---------|
| **第一层** | 长程司南（North Star）| 极小（<5%）| 第一卷规划时锁定 |
| **第二层** | 卷级大纲 | 中等（30%）| 提前2卷规划窗口 |
| **第三层** | 章节工作区 | 大（随时调整）| 前5章可完全改写 |

### 5.2 长程司南（第一层）

**定义**：全书的最终意境画面和核心信念，不绑定具体情节。

**正确示例**：
> 全书最终，林天站在废墟上看日出，他终于理解了父亲说的"力量是用来守护的"。

**错误示例**（太具体，会束缚中期创作）：
> 林天在第23卷第5章击败了大反派王朝并建立了新政权。

**司南的作用**：
- 每当规划新卷时，用司南校验：这一卷的发展方向是否朝着司南靠近？
- 当情节出现重大偏离时，用司南做仲裁

### 5.3 卷级大纲（第二层）

**规划策略**：
- 始终只提前规划 **当前卷+下一卷**
- 不对3卷后的情节做详细规划（"超期规划陷阱"）
- 每次开启新卷前，可输入**读者热力数据**来调整方向：
  - 章节热力图（哪些章节留存率高/低）
  - 读者评论摘要（哪些情节被讨论最多）
  - 跌落点分析（读者在哪章开始流失）

**卷级大纲可调节内容**（30%空间）：
- 调整子情节顺序
- 增删次要角色
- 改变某个反派的身份或阵营
- 调整爽感高潮的分布位置

### 5.4 章节工作区（第三层）

**定义**：当前正在写的章节 ± 前后5章的灵活调整空间。

**状态管理**：
- 写作完成的章节：**冻结**（不再修改，KB已录入）
- 当前写作的章节：**活跃**（可完全重写）
- 未来5章：**规划中**（有场景卡但可调整）

## 6. 八大创作法则

这是系统最核心的创作智慧，所有写作调用都内嵌这八条法则。

### 法则一：极限铺垫原则（四层困境）

**核心思路**：欲扬先抑——先把主角压到极限，再释放极限爽感。

**四层困境**（同时施压，不是选一个）：

| 层次 | 描述 | 示例 |
|------|------|------|
| **宏观困境** | 世界规则层面的压制 | 阶级制度、法律管控、天花板封印 |
| **中观困境** | 势力/团体层面的压制 | 门阀倾轧、组织背叛、多方围剿 |
| **微观困境** | 个人资源层面的压制 | 丹药耗尽、盟友背叛、情报误导 |
| **个人困境** | 心理/情感层面的压制 | 至亲受威胁、信仰崩塌、身份暴露 |

**执行要点**：
- 困境叠加的顺序：宏观→中观→微观→个人，逐层深入
- 每一层困境都必须看起来"无解"——读者要相信主角真的很惨
- 最后一层个人困境往往是"触发主角爆发"的导火索

---

### 法则二：期待感管理系统（信息差分层释放）

**核心思路**：不把信息一次性告诉读者，制造持续的"我想知道"欲望。

**三类钩子体系**：

1. **章末小钩子**（每章必须）：最后50-100字卡在关键动作之前，或揭示让读者意外的信息
2. **卷级大钩子**（每卷1-2个）：关乎主角命运的重大悬念，跨章维持期待
3. **全书超级钩子**（1-3个）：呼应长程司南，全书核心谜题

**信息差设计原则**：
- 让读者比主角**多知道一件事**（读者知道危险在哪，主角不知道）
- 或让读者比主角**少知道一件事**（主角有秘密，读者想猜）
- 两者交替使用，制造不同类型的紧张感

**九连环钩子设计（高级）**：
- 回收一个钩子的同时，埋下一个更大的钩子
- 确保读者始终有"还有一件事没解决"的感觉

---

### 法则三：连锁震惊反应（三层震惊）

**核心思路**：一件爽事发生时，让多个角色依次震惊，放大情绪涟漪。

**三层震惊结构**：

1. **直接执行者震惊**：被打脸的人的第一反应（不敢置信→愤怒/崩溃）
2. **周围亲友的震惊**：旁观者的反应（为主角欢呼/为反派叹气）
3. **权威人物的震惊**：高层/权威角色的惊讶（「连他也…？」）

**写法要点**：
- 三层震惊不要写成流水账，每层用不同的**感官细节**呈现
- 第三层权威震惊往往是最有爽感的：地位越高的人越惊讶，越爽
- 适当加入"旁白式总结"（类似弹幕效果：场内所有人都震惊了）

---

### 法则四：角色与剧情基因锚定（双轨弧光）

**核心思路**：男频小说有固定的男性基因驱动，所有角色设计都服务于此。

**主角行为准则**：主角的一切行为，其本质是"展示价值"，让世界（尤其是配角和对手）见证自己的强大。

**人物立体化三要素**：
1. **缺点设计**：主角必须有性格缺点（不是菜，是有执念或短板），这是成长的起点
2. **底层行为逻辑**：每个核心角色有一条"无论如何都不妥协"的底线，确保行为一致
3. **高概念生存动机**：主角的奋斗必须锚定一个有信仰底线的动机（家人安全、族群崛起、信念守护等）

**反派设计原则**：
- 反派的"恶"来自**立场冲突**，而非"天生坏人"
- 每个反派有专属的"攻击主角底线"的手段
- 反派智商必须在线——一个蠢反派提供不了真正的爽感

---

### 法则五：核心创作套路库

**解锁本法则**：在法则一到四的基础上，叠加以下经过商业验证的情节模式。

**套路一："锁"式危机设计**

- **时间锁**：设置倒计时压力（3分钟内/30天内/七天后），瞬间提升紧迫感
- **空间锁**：将角色限制在有限空间内（密室/孤岛/包围圈），强制制造冲突

**套路二："大小反差"设计**

- **大人物遇小事**：身份尊贵的主角处理日常小事，产生"降维打击"或"水土不服"的效果
- **小人物遇大事**：平凡主角卷入超规格事件，读者为其担忧和好奇

**套路三："道德挑战"设计**

- 设计主角在道德/利益/情感三者之间的两难选择
- 读者会极度投入地思考"我会怎么选"
- 主角的选择必须**符合角色底层逻辑**，而非随机的

---

### 法则六：数据化质量评估原则

系统的 8 维度质量评估，是法则六的实体化。见[第10节：质量评估系统](#10-质量评估系统)。

**核心优先级**：当质量评估中的维度与"爽感"产生冲突时，**爽感优先**。

---

### 法则七：高智商压迫与期待感拉扯

**核心公式**：没有极度憋屈，就没有极度爽快。

**反派设计要求（关键）**：

- 反派必须使用**规则杀（Rule Kill）**，而非纯暴力
  - 法规封口、阶层壁垒、信息管控、门阀制度……
  - 主角的胜利也必须是**对规则的瓦解**，而非单纯碾压
- 反派的计划必须**看起来天衣无缝**——读者要相信"这次主角真的赢不了"
- 反派智商不足→读者不觉得爽（因为赢得太容易）

**信息差陷阱设计**：

> 每隔5章必须设计一个"读者已知但主角未知"的信息陷阱。

类型：
- 反派的真实身份（读者知道，主角不知道）
- 即将到来的背叛（读者看到了迹象，主角还信任着那个人）
- 主角误解的关键线索
- 主角正在走入的埋伏

---

### 法则八：降维打击与错位震惊

**核心思路**：主角每次出手，必须打破常规认知——反派越自信，主角的反击越爽。

**降维打击四步结构**：

1. **反派布局完成**：反派认为已经赢定了，计划周密
2. **主角方形势最低落**：读者也开始为主角担忧（法则一的最深处）
3. **主角掀开金手指底牌**：用反派完全没料到的方式反击
4. **反派信仰崩塌**：反派的"怀疑人生"越深，读者越爽

**执行细节**：
- 降维打击不是"主角突然变强"——必须是**提前埋下的设定被激活**
- 反派的崩溃要写细：微表情、语气变化、动作细节（手抖、声音颤）
- 旁观者的"集体疑惑→震惊→服气"的过程要完整写出来

## 7. 流水线写作模式（Pipeline）

**适用场景**：稳定批量生产章节，无需用户即时介入。

### 7.1 架构总览

```
orchestrator (主调度进程)
  │
  ├─ 1. 读取 state.json + scene_plan
  │      提取当前章节的涉及角色IDs
  │
  ├─ 2. 切片 knowledge_base.json
  │      仅提取涉及角色/地点/事件的相关条目 → kb_slice
  │
  ├─ 3. 组装 instruction package
  │      = 场景卡 + kb_slice + 八大法则 + 风格规则 + 爽感类型
  │
  ├─ 4. 启动 chapter-writer sub-agent
  │      → 输出：章节正文 + kb_diff（JSON格式）
  │
  ├─ 5. 应用 kb_diff
  │      → 增量更新 knowledge_base.json
  │
  ├─ 6. 启动 summary sub-agent
  │      → 生成章节概括（约150字）
  │      → 更新 summary_index.md
  │
  ├─ 7. 质量评估
  │      → 8维度评分
  │      → 如 < 64分，触发 Critic sub-agent + 修订循环
  │      → 生成 quality_report
  │
  ├─ 8. 更新 state.json
  │      current_chapter += 1
  │      shuang_type_log 记录爽感类型
  │
  └─ 9. 向用户报告
         章节摘要 + 质量分数 + 下章预告
```

### 7.2 Chapter-Writer Sub-Agent 输出格式

chapter-writer 的输出是**双文件**：

**文件1：章节正文（Markdown）**
```markdown
# 第N章：[章名]

[正文内容，2000-4000字]
```

**文件2：KB差异文件（JSON）**
```json
{
  "chapter": "chapter_XX",
  "timestamp": "...",
  "add": {
    "entities": {
      "characters": {...},
      "locations": {...}
    },
    "foreshadowing": {...}
  },
  "update": {
    "entities.characters.char_001.arc_history": [...],
    "foreshadowing.planted.1.status": "resolved"
  }
}
```

### 7.3 Critic Sub-Agent（质量门控）

当某一章节质量评分 < 64分（合格线），自动触发 Critic sub-agent：

```
Critic sub-agent 流程：
  1. 读取原始章节 + 8维度评分详情
  2. 识别最低分维度（通常是 qd_07 钩子 或 qd_01 爽感）
  3. 生成具体修订指令（不是泛泛"写好点"，是精准操作）
  4. 将修订指令传回 chapter-writer 进行重写
  5. 重写后再次质量评估
  6. 最多循环 2 次（防止无限循环）
```

Critic 的 `tension_signal` 字段：
```json
{
  "tension_signal": {
    "current_level": 0.35,
    "trend": "falling",
    "recommendation": "立刻在场景中引入外部冲突"
  }
}
```

### 7.4 Cycle 收敛检查

每完成5章（一个cycle）后，自动执行收敛检查：

**检查项目**：
1. 爽感类型分布（5章中是否单调重复同一类爽感？）
2. 平均质量分数（本cycle均分 vs 上cycle均分）
3. 伏笔状态（新增伏笔数量 vs 回收伏笔数量，是否积压过多？）
4. 节奏分析（动作戏/对话戏/情感戏的比例）
5. 角色出场分布（是否某个重要角色消失太久？）

**收敛报告格式**：
```markdown
## Cycle 001 收敛报告

| 维度 | 本cycle | 目标 | 状态 |
|------|---------|------|------|
| 平均质量 | 68.2 | ≥64 | ✅ |
| 钩子命中率 | 4/5 | ≥4/5 | ✅ |
| 伏笔净植入 | +3 | ≤+5 | ✅ |
| 爽感类型多样性 | 3种 | ≥3种 | ✅ |
```

### 7.5 触发流水线写作

```
# 写单章
用叙事空间系统流水线写作第 [N] 章

# 写一个cycle（5章）
用叙事空间系统流水线写作 cycle 002，从第 [N] 章开始

# 继续写（自动读取state.json）
用叙事空间系统继续流水线写作
```

### 7.6 线热度连续性系统（Line Heat Continuity）

写作模块内置**线热度连续性规则**，防止长篇创作中故事线"消失"或"突然回收"。

**四类追踪对象**：

1. **角色线** — 核心与高频配角的出场节奏
2. **情节线** — 主线/支线的推进状态
3. **伏笔线** — 待回收的置入承诺
4. **算法标签线**（番茄特化）— 金手指热度、主线驱动力、书名关键词呼应频率

**热度状态**：

| 热度 | 含义 |
|------|------|
| `hot` | 当前正在驱动冲突或读者期待 |
| `warm` | 近期活跃，但非本章主焦点 |
| `cold` | 若再忽略将造成遗忘或断线风险 |
| `archived` | 已明确收束或长期休眠 |

**默认回顾阈值**：
- 核心角色：3-5章内需有有效触及
- 主要支线：4-6章内需有推进、转移或明确延迟
- 活跃伏笔：连续错过2次可回收窗口即判 overdue

**有效触及 vs 无效露面**：

| 类型 | 判定 |
|------|------|
| 直接推进该线 | ✅ 有效 |
| 压力转移并形成新义务 | ✅ 有效 |
| 给出可验证的延迟理由 | ✅ 有效 |
| 只点名不改变局面 | ❌ 无效 |
| 只做回忆性提及无后效 | ❌ 无效 |

**算法标签**（高追读率线条附加标注）：

| 标签 | 含义 |
|------|------|
| `#日更` | 本线近期驱动过连续更新承诺 |
| `#爆更点` | 本线即将触发大量信息差释放或连续爽点 |
| `#读者悬念` | 本线当前处于"读者已知而主角未知"状态 |
| `#深坑待填` | 本线有超过10章未回收的伏笔，进入高优先级 |

> `#深坑待填` 超过15章自动触发 WARN：伏笔遗忘风险。

**章后写回要求**（每章结束时记录）：
1. 本章触及了哪些线
2. 各线热度是否变化
3. `next_review_chapter` 是否需要前移
4. 是否产生 overdue 项

## 8. 互动跑团模式（Interactive）

**适用场景**：用户想实时扮演主角，对关键决策有掌控权的沉浸式创作体验。

### 8.1 角色分工

| 角色 | 由谁扮演 | 职责 |
|------|---------|------|
| **DM（地下城主）** | AI | 世界呈现、环境描述、NPC 言行、事件触发 |
| **玩家（主角）** | 用户 | 主角的关键决策、对话内容、行动选择 |

### 8.2 状态机（全流程）

```
INIT ──→ OPENING ──→ PING_PONG ←──→ ROLLBACK
                         │
                         ↓ (节奏报警)
                   PACING_ALERT
                         │
                         ↓ (手动或触发落盘)
                      LANDING ──→ MAINTENANCE
```

**各状态说明**：

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| **INIT** | 初始化状态 | 用户启动跑团模式 |
| **OPENING** | 开局叙述 | INIT 完成后 |
| **PING_PONG** | 主循环：DM描述+玩家行动 | OPENING 完成后 |
| **ROLLBACK** | 回滚到上一个决策点 | 用户输入"撤销"/"重来" |
| **PACING_ALERT** | 节奏警告 | 连续3回合无实质推进 |
| **LANDING** | 本章落盘（保存）| 用户输入"落盘"或达到字数上限 |
| **MAINTENANCE** | 维护/KB同步 | LANDING 完成后 |

### 8.3 PING_PONG 主循环

每一回合的标准格式：

**DM 输出（AI）**：
```
---
[环境/状态更新：2-3行，简洁]

[场景核心描述：50-150字，使用番茄文风格：短句、画面感]

[NPC 言行（如有）]

[选项提示（可选）]：
A. [行动选项A]
B. [行动选项B]
C. 自定义行动
---
```

**玩家输入（用户）**：
- 选择选项（A/B/C）
- 或直接输入自定义行动

**每回合字数目标**：300-600字（DM部分），不超过600字，保持快节奏

### 8.4 ROLLBACK（回滚机制）

当玩家对某个回合结果不满意：

```
用户输入：撤销 / 重来 / rollback
```

系统执行：
1. 恢复到上一个决策点的状态
2. 保留KB中已确认的实体（不回滚）
3. 重新提供该决策点的选项（可增加新选项）

**回滚限制**：最多回滚3次，超过后提示用户"本段剧情已锁定"。

### 8.5 LANDING（落盘）

落盘 = 本次跑团的成果保存为正式章节文件。

**触发方式**：
```
用户输入：落盘 / 结束本章 / landing
```

**落盘流程**：
1. DM 整理本回合所有片段
2. 生成完整章节正文（修复格式、补全场景衔接）
3. 生成 kb_diff（记录本章的实体变更）
4. 应用 kb_diff 到 knowledge_base.json
5. 生成章节概括，更新 summary_index.md
6. 更新 state.json
7. 生成质量报告

### 8.6 PACING_ALERT（节奏警告）

触发条件：
- 连续3回合玩家只进行对话，无实质行动推进
- 连续3回合场景没有发生变化
- 累计字数超过3000字但剧情推进不足0.5个场景卡节点

警告格式：
```
⚠️ 节奏提醒：当前剧情推进较慢。
建议：引入一个外部事件（袭击/消息/意外），
或选择"跳过"当前互动，直接推进到下一个关键节点。
```

### 8.7 互动模式触发

```
# 启动互动模式
用叙事空间系统启动跑团模式，我扮演主角[名字]

# 指定从某章开始
用叙事空间系统跑团模式从第[N]章的场景开始

# 落盘保存
落盘

# 回滚
撤销 / 重来
```

### 9.4 KB Diff 机制

每章写作完成后，chapter-writer 自动输出一个 diff 文件到 `.xushikj/kb_diffs/`，由 orchestrator 增量应用到 `knowledge_base.json`。

**Diff 文件完整格式**：

```json
{
  "chapter": 7,
  "changes": {
    "characters": {
      "char_001": {
        "update": { "status": "受伤", "arc_stage": "挾扎期", "last_seen_chapter": 7 },
        "append": { "abilities": ["暗影感知（覚醒）"] }

        }
      }
    },
    "locations": {
      "loc_003": { "update": { "current_state": "废墟" } }
    }
  },
  "relationships": {
    "append": [
      {
        "entity_a": "char_001",
        "entity_b": "char_new_001",
        "type": "敌对",
        "evolution_log": [{"chapter": 7, "status": "初次交锋"}]
      }
    ],
    "evolve": [
      {
        "entity_a": "char_001",
        "entity_b": "char_002",
        "add_log": {"chapter": 7, "status": "信任加深，共同作战"}
      }
    ]
  },
  "timeline_append": [
    {
      "chapter": 7,
      "event": "暗影魔突袭学园",
      "entities_involved": ["char_001", "loc_003"],
      "consequences": "学园东翂损毁，char_001 负伤"

        "id": "foreshadow_008",

```

**Diff 操作语义**：

| 操作 | 含义 | 适用节点 |
|------|------|----------|
| `update` | 覆盖指定字段（merge，不替换整个对象）| entities 实体对象内 |
| `append` | 追加到数组字段（不去重）| entities 数组字段 |
| `create` | 新建实体（必须包含模板全部字段）| entities 新 ID |
| `evolve` | 向已有关系的 `evolution_log` 追加记录 | `relationships.evolve` |
| `timeline_append` | 追加到 timeline 数组 | 顶层 |
| `plant` | 新增伏笔到 `foreshadowing.planted` | `foreshadowing` |
| `resolve` | 将伏笔 ID 移入 resolved，标记已回收 | `foreshadowing` |

## 10. 质量评估系统

### 10.1 八维度评分体系

| 编号 | 维度名称 | 满分 | 合格线 | 核心问题 |
|------|---------|------|-------|---------|
| qd_01 | 爽感与情绪反馈 | 10 | 8 | 主角行为是否带来强烈正向反馈？ |
| qd_02 | 金手指利用率 | 10 | 8 | 本章是否有效展示了金手指？ |
| qd_03 | 节奏与信息密度 | 10 | 8 | 剧情是否推进迅速、无废话？ |
| qd_04 | 角色对话独特性 | 10 | 8 | 对话是否鲜明反映各角色性格？ |
| qd_05 | 角色塑造一致性 | 10 | 8 | 角色言行是否始终符合设定？ |
| qd_06 | 意境与主题匹配 | 10 | 8 | 场景描写是否服务于整体主题？ |
| qd_07 | 章末悬念（钩子）| 10 | **7（硬门槛）** | 结尾是否有强烈翻页欲？ |
| qd_08 | 语言下沉度 | 10 | 8 | 用词是否白话易读、短句占比高？ |
| **合计** | | **80** | **64** | |

### 10.2 qd_07 硬门槛

**特别注意**：qd_07（章末钩子）有硬门槛设定：

> **低于 7 分触发 WARNING**，自动进入修订循环

章末钩子在商业网文中是"留住读者的唯一工具"，哪怕全文质量很好，没有好钩子读者也会弃坑。

**优秀钩子的特征**：
- 卡在关键动作**之前**（读者不得不往下看才知道结果）
- 揭示一个意外信息（让读者"咦？"）
- 制造新的问题（而非回答已有问题）

**差钩子的特征**：
- 章节事件圆满收束（没有悬念）
- 钩子只是情绪性的（"他感到一阵恐惧"——不具体）
- 钩子与下章内容相关性弱

### 10.3 质量评估触发

**自动触发**：每次流水线或互动落盘后自动执行

**手动触发**：
```

```

```markdown

| qd_02 金手指 | 9/10 | 棋局预测能力成功展示 |
| ...  | ...  | ... |
| **总分** | **68/80** | ✅ 合格（64分线） |

### 改进建议
- qd_03 节奏偏慢：第3段有2段环境描写可压缩
- qd_07 钩子较弱：建议改为卡在主角发现匕首的瞬间
```

## 11. 帮回辅助系统

### 11.1 系统说明

帮回系统是**独立于写作模式**的即时辅助机制：无论当前在流水线模式还是互动模式，用户随时可以输入帮回指令获得叙事策略支持。

**系统架构**：
- 8种即时行动指令（甲类）
- 1种章节规划指令（乙类）
- 2种分析诊断指令（丙类）

### 11.2 八种甲类指令

| 指令 | 类别 | 效果 |
|------|------|------|
| `帮回主动1` | 主动果断 | AI 生成主角果断行动/宣告的角色内片段 |
| `帮回主动2` | 主动策略 | AI 生成主角智慧性、策略性的主动出击片段 |
| `帮回被动1` | 被迫顺从 | AI 生成主角迫于外部压力退让/隐忍的片段 |
| `帮回被动2` | 情感依赖 | AI 生成主角因内在情感疲惫、需要慰藉的片段 |
| `帮回黑暗1` | 心理掌控 | AI 生成主角心理层面的掌控欲/操控/执念描写（极致微表情）|
| `帮回黑暗2` | 行动侵略 | AI 生成主角直接、具占有性的行为描写（遵循内容规范）|
| `帮回推进1` | 宏大旁白 | AI 生成第三方旁白，用于关键转折或渲染宏大叙事 |
| `帮回推进2` | 细节旁白 | AI 生成第三方旁白，深化感官细节/微表情/环境互动 |

### 11.3 帮回甲类的两种执行模式

**默认模式——直接演绎**：
1. AI 直接输出主角行动/对话（角色内）
2. 紧接着 AI 代入 NPC 做出角色内回应
3. 无需用户再次输入，场景连贯推进

## 12. 去AI痕迹后处理（Humanizer）

### 12.1 核心原则

```
只做减法和替换，不加新内容。
保护小说叙事：象征、隐喻、心理描写、破折号都是合法文学工具，不动。
宁可漏改，不可误伤。
```

### 12.2 规则一：「不是…而是…」句式清除

**扫描目标**（所有否定+转折对比句）：
- `不是A，而是B`
- `不是A，是B`
- `不仅是A，更是B`
- `与其说是A，不如说是B`

**清除策略（99%删除率）**：

只有当同时满足以下两条件时才保留原句：
1. 删除会导致剧情逻辑断裂（不是修辞问题，是信息缺失）
2. 对比双方存在读者无法自行推导的信息差

**四种重写方式**（按优先级）：

| 方式 | 示例 |
|------|------|
| 转为单肯定 | `不是勇敢，而是绝望` → `那是绝望` |
| 转为双肯定 | `不是冷漠，而是克制` → `是克制，也是自我保护` |
| 转为具体描写 | `不是恐惧，而是敬畏` → `他的膝盖不自觉地弯了下去` |
| 直接保留B | `不是风声，而是哭声` → `那是哭声` |

### 12.3 规则二：稀疏排版/极短句清除

**扫描目标**：
- 独占一行且 ≤15 字的非对话句子
- 连续 2+ 个短句各占一行（诗歌体排版）
- 段落只有 1-2 句话（非场景转换标记）

**处理策略**：
- 合并到前段末尾，用逗号/句号衔接
- 语义独立的短句：重写为完整句子融入上下文

**例外（不动）**：
- 对话中的自然停顿
- 特殊格式（章节标题、节奏极快的打斗动作）
- 场景切换标记（`——三天后——`）

### 12.4 规则三：高频AI词汇替换

核心替换表（部分示例）：

| 原文（AI痕迹）| 替换方向 |
|------------|---------|
| 此刻 | 这一刻 / 现在 / 当下 |
| 然而 | 但 / 只是 / 偏偏 |
| 同时 | 顺带 / 顺手 / 一边……一边…… |
| 不仅如此 | 删除，直接说后半句 |
| 令人窒息 | 他胸腔发紧 / 气堵在喉咙里 |
| 从而 | 直接写结果，删掉连接词 |
| 其实 | 尽量删除或改为具体化陈述 |

**Humanizer 的操作方式**：

```
# 对一段文本进行处理
用叙事空间系统对以下文本去AI痕迹处理：
[粘贴文本]

# 对整章进行处理
用叙事空间系统对第[N]章进行 humanizer 处理
```

## 16. 常见问题FAQ

### Q1：没有安装 MCP 工具，系统还能用吗？

**可以**。MCP 工具是增强功能，非必须。降级处理：
- Step 0 对标分析：跳过，或用户手动提供对标作品的分析资料
- Step 2.5 世界观：由用户直接输入世界观设计，系统不调用 websearch
- 大纲推演（Steps 4/6/8-9）：使用内置的线性推演替代，深度略低

### Q2：如何从第 N 章继续写，而不是从头开始？

确保 `.xushikj/state.json` 的 `writing_state.current_chapter` 已更新到正确的章节号，然后：

```
用叙事空间系统继续写作，从第[N+1]章开始
```

系统会自动读取 state.json 和 summary_index.md 恢复上下文。

### Q3：如何开始第二卷？

```
用叙事空间系统开启第2卷规划
```

系统自动执行：
1. `state.json.planning_state.current_volume` → 2
2. `state.json.planning_state.current_step` → 回退到 Step 4（卷大纲规划）
3. 提示用户：是否输入第一卷的读者热力数据来调整第二卷方向？

### Q4：quality_report 显示某章得分很低怎么办？

系统会自动触发 Critic sub-agent 进行修订。如果自动修订后仍不合格：

```
用叙事空间系统重写第[N]章，重点提升 qd_07（钩子）
```

手动指定要提升的维度，系统会针对性地重写该部分。

### Q5：跑团模式落盘后如何切回流水线模式？

```
用叙事空间系统切换到流水线模式，从下一章开始
```

系统更新 `state.json.writing_mode` 为 `pipeline`，并基于落盘的章节内容更新 scene_plan。

### Q6：伏笔积压太多（pending 数量超过 10 个）怎么办？

触发帮回指令：
```
帮回爽点分析
```

或手动触发伏笔回收规划：
```
用叙事空间系统分析当前积压伏笔，规划最近5章的回收顺序
```

### Q7：角色行为出现前后矛盾怎么办？

```
帮回人设检查 [角色名]
```

系统会对比该角色的初始设定（Step 3）、KB 中的 arc_history 和最近的实际写作内容，找出矛盾点并提供修复建议。

### Q8：生成的文本 AI 感很强，如何处理？

```
用叙事空间系统对第[N]章进行 humanizer 处理
```

或粘贴具体文本段落请求处理。Humanizer 专门针对番茄文的 AI 痕迹清除，不会改动场景结构。

### Q9：想临时更改世界观或金手指设定怎么办？

1. 更新 `.xushikj/outline/worldview_and_system.md`
2. 更新 `.xushikj/knowledge_base.json` 中受影响的 abilities 实体
3. 告知系统：
   ```
   用叙事空间系统：我更新了世界观，已修改 worldview 文件，请同步理解后继续
   ```

### Q10：写到一半想换成另一个主角视角怎么办？

这属于重大方向调整，建议：
1. 先执行落盘（保存已有内容）
2. 更新 Step 3 的人物介绍（调整双轨弧光规划）
3. 更新 `.xushikj/outline/character_arcs.md`
4. 重新执行 Step 5（人物大纲）
5. 告知系统从哪章开始改变视角

| `S5` | 收获型 | 主角获得重要资源/盟友/信息 | 法则二（期待感）|
| `S6` | 智计型 | 主角用智谋解决强敌 | 法则七（高智商压迫）|
| `S7` | 成长型 | 主角能力/心性的质变节点 | 法则四（角色弧光）|
| `S8` | 情感型 | 深情互动/羁绊确立 | 法则四（男性基因）|

*每个 cycle（5章）中，S1-S8 应有至少3种类型出现，避免单调重复。*

## 附录 B：番茄文风格快速对照表

| 避免 | 使用 |
|------|------|
| 长段落环境描写（>5句）| 1-2句点到为止，突出画面感 |
| 复杂从句 | 拆成短句，每句≤20字 |
| 情感直陈（"他感到害怕"）| 感官/动作具象化（"他的手不受控制地颤了"）|
| 四字成语堆砌 | 用口语化的动词/动作替代 |
| "不是A而是B" | 直接说B（见第12节）|
| 视角跳跃（同段多人视角）| 固定单一视角，每段只有一个感知主体 |
| 过度解释反派动机 | 让读者自行推断，只写表象 |
| 平铺直叙回忆 | 化为现在时的感官闪回（≤3句）|
